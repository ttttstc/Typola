# AI Diff Preview SPEC —— 整篇改稿分段对比 + oneshot 复用现有卡

- **日期**:2026-06-23
- **目标**:任何 AI 生成的「整篇/多段 markdown」在写回原文前,**先按段对比 + 逐段采纳/拒绝**,避免一覆盖就丢原文。
- **不动**:选区 oneshot(润色/缩写/扩写/校对/解释)的 `SelectionResultCard`,已经满足「先对比再应用」体验。

---

## 一、要解决的痛点

| 场景 | 现状 | 痛在哪 |
|---|---|---|
| 选区 oneshot(润色/缩写等) | ✅ 浮卡显示原文 vs 新版,「采纳/取消」 | — 已 OK |
| 检视模式「发 AI 改稿」 | ❌ AI 改完整篇 → 落 `.typola-output/xxx.md` → 用户**自己打开新文档肉眼对比** | 完全没 diff,跟"直接覆盖"差不多 |
| Skill OS skill 产物 | ❌ 同上,产物文件靠用户手动 diff | 同上 |
| 未来 AI 整篇生成 / 多段改 | ❌ 没有标准化的对比层 | 每加一类 AI 入口都得自己造 diff |

## 二、设计

### 2.1 什么时候进审阅态(触发判定 —— 必须严格,不靠意图检测)

**核心边界:审阅态从不被「自由对话」自动触发。** 判定只看「入口是否声明了『输出 = 当前文档的修订版』+ 有没有原文基线」,不去猜 AI 到底想干嘛。三类:

| 类 | 行为 | 入口 |
|---|---|---|
| **A. 直接进审阅态** | 入口已**声明**"输出是当前文档的修订版" + 有原文基线 → 立刻进内联审阅 | • 检视模式「发 AI 改稿」(固定意图)<br>• Skill frontmatter 声明 `output: review_inline`(如"全文重写/全文润色")<br>•(未来)命令面板「改进全文 / 重写全文」 |
| **B. 先落产物,审阅是显式动作** | 开放意图或产物类 → 先落右侧 artifact;产物卡带「合并到当前文档」按钮,**点了才**用 (产物 vs 当前文档) 进审阅态 | • **左侧栏自由对话**(无论 AI 生成什么,绝不自动碰当前文档)<br>• Skill `output: new_doc` |
| **C. 不进审阅态** | 走现成路径 | • 选区 oneshot(润色/缩写等)→ `SelectionResultCard` 局部对比卡(不是全文审阅态)<br>• 海报 / HTML / 全新独立文档 → artifact 直接打开(无基线,不审阅) |

**判定的唯一信号源**:触发入口的类型 / skill 的 `output` 声明 / 用户在产物卡上的显式点击。**不解析 AI 输出内容来猜意图。**

### 2.2 统一入口

A 类、B 类(点了合并)最终都调同一个函数:

```ts
openDiffReview({
  originalContent: string,   // 原文档全文(基线)
  proposedContent: string,   // AI 给的新版本 / 产物内容
  source: 'review' | 'skill' | 'merge-artifact',  // 触发来源(用于顶部标题)
  title?: string,            // 可选:"检视意见已应用" / "合并:周报草稿" 等
  onApply: (mergedContent: string) => void,  // 用户处理完,合并后回写当前文档
  onCancel: () => void,
});
```

接入点:
- **检视模式「发 AI 改稿」**(`ReviewSidebarPanel.tsx`,现 `handleSendReviewToAI`)产物到达 → **不开新文档**,直接 `openDiffReview`(A 类)
- **自由对话 / skill 产物 chip** → 卡上加「合并到当前文档」按钮,点击才 `openDiffReview`(原文 = 当前打开的文档,proposedContent = 产物内容)(B 类)
- **声明 `output: review_inline` 的 skill** → 产物到达直接 `openDiffReview`(A 类)

### 2.3 UI 形态 —— 文档内联审阅 + 导航三件套

**不是右侧面板,是把中间编辑区临时切成「审阅态」**:用我们自己渲染的**只读 diff 视图**覆盖正常的 Vditor 编辑区,在原文里行内标出每处改动(原文 strikethrough + 新版高亮),用户在上下文里逐处采纳/拒绝。

> **为什么能做行内**:F-A1 调研说 Vditor 不支持行内装饰层 —— 但那个限制只针对**正在编辑的 live Vditor**(它会 outerHTML 整块重写擦掉 ghost DOM)。审阅是**只读时刻**,用户不在敲字,我们渲染自己的只读 DOM,完全绕开 Vditor 的擦写。用户点「应用」后才把合并结果写回 Vditor。

```
┌─ 周报.md ──────────────────────────────────────────┐
│ ◇ 审阅 AI 改动   ▲ 第 3/8 处 ▼   [全部采纳][全部拒绝][应用] │  ← 顶部审阅条
├──────────────────────────────────────────────────┤▕
│ 本周工作小结                                          │▕  ← 右侧滚动条
│ ~~本周完成了三个主要的任务项目~~ 本周完成三项任务,…     │▕     变成「改动地图」:
│ 上线评审 ~~由于种种原因导致~~ 导致进度延后,下周补上。   │▕  ·  8 个标记 = 8 处改动
│ ┌────────────────────────────────────────┐       │▕  ●  当前处高亮放大
│ │第三方接口 ~~存在稳定性问题~~ 稳定性待观察  [✓][✗]│←当前│▕  ·  点标记跳到该处
│ │  Y 采纳 · N 拒绝 · ⏎ 下一处                  │       │▕  ·
│ └────────────────────────────────────────┘       │▕  ·
│ 团队协作 +效率有所提升,沟通更顺畅。                    │▕  ·
└──────────────────────────────────────────────────┘▕  ·
```

**导航三件套**(解决「内联怎么快速找到所有改动」):

1. **计数器 + 上/下一处 + 键盘**:顶部 `第 3/8 处` + `▲▼` 按钮,点一下跳到下/上一处并滚到屏幕中央。键盘 `Y` 采纳 / `N` 拒绝 / `⏎` 跳下一处,可全程不碰鼠标审完(像 `git add -p`)。
2. **滚动条改动地图**:右侧滚动条变成标记条,N 处改动 = N 个小标记,当前处高亮放大;点任一标记跳到对应改动。给「全局分布感」+ 快速跳转。
3. **全部采纳 / 全部拒绝**:不想逐个看就一键收尾。

**默认状态**:每处 diff **默认采纳**(用户最常意图是大部分都用 AI 的),逐处可点「拒绝」翻转。顶部 `应用` 把当前所有处的状态合并成 mergedContent 写回原文,关闭审阅态回到正常编辑。

**不做**(本期):折叠式「改动总览抽屉」(把所有改动摘要列成一个可展开列表)。导航三件套对改动 < 15 处的文档已够用,真嫌不够再加,避免重新引入右侧面板的常驻占地。

### 2.4 分段算法

**分段单位**:markdown 段落(空行 `\n\n` 分隔)。理由:段落是 markdown 最自然的语义单位,标题/列表/代码块都是独立段。

**diff 计算**(纯前端,不引 Rust):
1. 把 original 和 proposed 各按 `\n\n+` 切成段数组
2. 用 LCS(最长公共子序列)算法对齐,得到段对(`orig` / `new` / 'both')
3. 三种 op:
   - `unchanged`:段一样 → 不显示(优化噪音)
   - `modified`:段对应但内容不同 → 显示"原 vs 新"
   - `added` / `removed`:只在一边出现 → 标 + / -

**依赖**:用 `diff` npm 包(轻量,~10KB)或者自己实现 LCS(代码 ~50 行)。不需要 `diff-match-patch`(过于强大)。

### 2.5 合并写回

逐个 diff 块走完后,组装 mergedContent:
- `unchanged` 段:原文保留
- `modified`:用户「采纳」→ 新版段;「拒绝/跳过」→ 原文段
- `added`:用户「采纳」→ 插入到对应位置;「拒绝」→ 不插入
- `removed`:用户「采纳删除」→ 不保留原段;「拒绝(保留)」→ 保留原段

写回入口:调编辑器 `setValue(mergedContent)` 或在源码模式下 `replaceAll`。文档自动标 dirty,用户自己保存。

### 2.6 oneshot 的"先对比再应用"

现状 `SelectionResultCard` 已经是这套语义,不动。**作为补充**:
- 把卡片的「采纳替换」按钮文案改成「应用更改」(对齐"先对比再应用"语义)
- 这一条 5 分钟,顺手做

### 2.7 审阅态生命周期与未保存逻辑

**核心原则:审阅态 ≠ 文档脏。** 审阅期间编辑器内容和磁盘都还是**原文**,AI 改动只活在临时 UI 状态(原文 + AI 新版 + 逐处采纳/拒绝决定)里。**只有点「应用」才把合并结果落进编辑器 → 文档此刻变脏 → 之后完全走 Typola 现有未保存逻辑**(`confirmUnsavedChoice` 三按钮对话框,任务 #14)。不为审阅态发明新的"未保存"概念。

**两条兜底保证**:
1. **原文从不丢**:审阅期间磁盘是原文,任何意外退出文档数据都在。
2. **AI 产物不丢**:每个 diff 入口的 AI 产物落在 `.typola-output/<conversation>/xxx.md`(检视发AI改、skill产物都有)。退审阅只丢「逐处采纳决定」(廉价),产物文件还在,从 chip 可重新进 diff。**约束:每个 diff 入口必须有可恢复的产物源(磁盘文件),否则退出会丢 AI 输出。**

**各退出场景**:

| 退出动作 | 审阅态(未点应用) | 已点应用后(文档已脏) |
|---|---|---|
| 点关闭 / Esc | 放弃审阅,恢复原文。**仅当用户动过 hunk** 才轻确认「放弃本次 AI 审阅?」;产物仍在可重开 | 已回正常编辑,N/A |
| 切 tab / 切文档 | 先退审阅(丢 pending)再切;产物可重开 | 现有未保存三按钮对话框 |
| 关文档 | 同上 | 现有未保存对话框 |
| 关 App / 窗口 | 直接退(文档没脏,**不弹保存**);产物可重开 | 现有未保存对话框 |
| 原文件被外部改 | 退审阅 + 提示「原文档已变,请重新发起」(不跨外部改动合并) | 走现有外部变更冲突逻辑 |

**应用写回**:`setValue(mergedContent)` + 同步压 AI 撤销栈 `{before: 原文, after: 合并版, selRange: null}`,应用后一次 Ctrl+Z 整体回退。autosave 开启时应用后会被自动保存(应用是显式动作,自动保存合理)。

- 顶部计数器显示 `第 N/总数 处`;`▲▼` 按钮 + 键盘 `Y/N/⏎` 能逐处跳转 + 采纳/拒绝
- 右侧滚动条改动地图:每处一个标记,当前处高亮,点标记跳到对应改动
- `全部采纳`/`全部拒绝` 一键收尾
- 点「应用」→ 按各处状态合并 → 写回当前文档 → 退出审阅态回正常编辑 → 文档标 dirty
- Skill OS 产物 chip 上「以 Diff 查看」按钮 → 用产物 vs 原文跑 diff
- 一处都不采纳 → 原文不变;全部采纳 → 等于直接覆盖
- 改动 20+ 处的长文档:滚动条地图 + 上/下一处导航能让用户快速过完每一处,不漏

## 五、不做

- ❌ **行内 ghost block 叠在 live Vditor 上**(Markra 的 ProseMirror DecorationSet 风格)——live Vditor 会 outerHTML 重写擦掉 ghost,精度脆。我们走**独立的只读审阅态 DOM**(只读时刻不受 Vditor 擦写约束),不是叠在编辑器上
- ❌ **折叠式改动总览抽屉**——导航三件套对 < 15 处改动够用,真不够再加
- ❌ **字符级 / 行级 diff**——段落级足够,用户不需要"这个字改了"那种精度
- ❌ **3-way merge**——AI 改稿是 2-way,够了
