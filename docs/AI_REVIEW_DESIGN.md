# AI 检视模式持续改稿与候选稿对比方案

## Problem Statement

当前 AI 文档改稿容易在三个方面失控：改稿过程与文档编辑割裂；每次反馈都重新启动对话并重复读取全文；AI 结果直接覆盖正文后，用户难以确认变化、继续演进或恢复原稿。

用户需要的是一条连续但受控的改稿链路：AI 能够在同一个对话中持续理解目标并迭代候选稿，用户能够通过直观 Diff 看清变化，正式文档只有在确认后才发生变化。

## Solution

将 AI 改稿放入现有检视模式，使用一个持续复用的 Provider-bound Conversation 作为 Session。Session 绑定当前文档，但不管理文档分支或派生关系。

Session 只维护三类状态：

- Conversation Context：通用讨论目标、写作偏好和前序结论；
- Active Document Context：当前源文档、版本快照和当前文档上下文；
- Revision Candidate：尚未写入正式文档的候选稿、Diff 和改稿状态。

一次普通改稿执行一轮“检视 → 修改 → 自检”，高影响改稿在修改前先展示影响范围。Agent 停在候选稿阶段，用户在左右对比界面中审阅、手动编辑、接受或拒绝修改，最后选择应用、另存或放弃。

完整链路如下：

```text
检视模式发起改稿
→ 同一 Session 检视、修改、自检
→ 生成 Revision Candidate
→ 左侧基线与右侧候选稿对比
→ 用户反馈或直接编辑候选稿
→ Agent 基于当前候选稿继续处理
→ 应用、另存或放弃
```

## User Stories

1. As a 文档作者, I want to 在当前检视模式中直接发起 AI 改稿, so that 我不需要切换到另一个工作台完成同一件事。
2. As a 文档作者, I want to 用自然语言描述改稿目标, so that 我不需要编写完整的系统 Prompt。
3. As a 文档作者, I want to 对选中文本、当前章节或全文发起改稿, so that 修改范围符合当前工作上下文。
4. As a 文档作者, I want to 普通改稿直接生成候选稿, so that 句子润色和压缩不会增加不必要的确认步骤。
5. As a 文档作者, I want to 在全文重写等高影响改稿前看到预计影响范围, so that 我可以在大范围修改发生前确认方向。
6. As a 文档作者, I want to 在同一个对话中连续提出多轮修改要求, so that Agent 能理解“再自然一点”“第三章不要动”等上下文。
7. As a 文档作者, I want to 切换到另一个文档时保留通用讨论上下文, so that 我不必重复说明长期写作偏好和主题背景。
8. As a 文档作者, I want to 切换文档后将新文档作为当前源内容加入上下文, so that 后续改稿针对的是新文档而不是旧文档。
9. As a 文档作者, I want to 在切换文档前收到未应用候选稿提示, so that 我不会意外丢失正在审阅的修改。
10. As a 文档作者, I want to 看到修改前后左右对比, so that 我能直观看出 AI 到底改了什么。
11. As a 文档作者, I want to 在右侧候选稿中直接手动修改, so that 简单词句调整不需要再次调用 AI。
12. As a 文档作者, I want to 接受或拒绝单处修改, so that 我可以精细控制候选稿内容。
13. As a 文档作者, I want to 继续针对当前差异、当前章节或全文给 Agent 反馈, so that 后续修改范围清晰可控。
14. As a 文档作者, I want to 在候选稿手动编辑后决定是否重新检视, so that 系统不会未经同意自动消耗模型调用。
15. As a 文档作者, I want to 在 AI 输出后看到简短的自检结果, so that 我知道本轮是否超出范围或产生了高影响变化。
16. As a 文档作者, I want to 在应用前始终保留正式文档不变, so that 我可以放心审阅候选稿。
17. As a 文档作者, I want to 将候选稿应用到当前文档并自动生成历史版本, so that 我可以继续修改且随时回退。
18. As a 文档作者, I want to 将全文重写结果另存为新文档, so that 原始文档保持不变，同时可以继续演进新稿。
19. As a 文档作者, I want to 另存为新文档后自动切换到新文档, so that 我可以在同一个对话中继续修改新稿。
20. As a 文档作者, I want to 从历史版本进入同一套 Diff 审阅流程, so that 恢复旧稿不会直接覆盖当前内容。
21. As a 文档作者, I want to 在应用前取消改稿, so that 当前正式文档完全不受影响。
22. As a 文档作者, I want to 关闭应用后恢复当前对话和候选稿, so that 长时间改稿不会因为退出程序而丢失。
23. As a 文档作者, I want to 在正式文档被外部修改后看到候选稿过期提示, so that 我不会把旧基线上的修改错误应用到新内容上。
24. As a 文档作者, I want to 选择以最新文档重新开始或保留候选稿人工对比, so that 外部修改不会被系统自动合并或覆盖。

## Implementation Decisions

- 改稿入口复用现有检视模式，不新增独立的改稿工作区。
- 一个 Provider-bound Conversation 对应一个持续 Session；同一 Session 内的检视、修改、复核和追问不重复创建新会话。
- Session 的通用讨论上下文在切换文档时保留；当前文档上下文、候选稿、Diff 和改稿反馈在切换时清空并重新建立。
- 切换文档后，新文档成为 Active Document Context 的源内容，并建立新的候选稿基线。
- 当前文档被外部修改时，将候选稿标记为过期；不自动合并。用户可以用最新文档重置候选稿，或保留候选稿并以最新文档为基线重新对比。
- Revision Candidate 是临时工作状态，不作为派生文档、分支或独立文档管理。
- 普通改稿执行一轮“检视 → 修改 → 自检”；Agent 默认在一轮完成后停止，用户的下一条反馈触发下一轮。
- 高影响改稿包括全文重写、章节合并、删除章节、结构重组和明显改变结论的修改。此类操作先展示影响范围，再由用户确认生成候选稿。
- 左侧对比区域显示修改前源内容，只读；右侧显示候选稿，可直接编辑。
- 接受或拒绝差异只修改候选稿，不修改正式文档。
- 候选稿手动编辑后，自检状态标记为过期；系统不自动重新调用 Agent，用户可以主动重新检视。
- 改稿反馈的默认作用范围跟随当前焦点，并支持当前差异、当前章节和全部候选稿三个范围。
- 自检只检查本轮要求是否完成、是否超出范围、是否产生大范围结构变化和是否存在高影响修改。不建立默认的受保护内容模型；用户明确提出的保留要求只在本轮有效。
- 自检结果分为通过、警告和阻断。阻断结果不能直接应用，警告结果可以进入 Diff 审阅。
- 应用候选稿到正式文档时，先保存应用前版本，再写入候选稿；候选稿变化本身不生成正式历史版本。
- 另存为新文档时，源文档保持不变，新文档自动成为当前文档，候选稿清空并以新文档内容建立新基线；当前 Session 继续复用。
- 历史版本恢复先进入候选稿和 Diff，用户确认应用后才覆盖当前文档。
- 关闭应用后恢复 Provider-bound Conversation、当前文档上下文、候选稿、Diff 和自检状态。
- 现有差异计算、候选合并和文档事务能力作为底层实现接缝；本方案新增的是 Session 与文档改稿状态之间的编排，不重新实现基础 Diff。

## Testing Decisions

测试以用户可观察行为为准，不测试模型内部推理过程或具体组件实现。

需要覆盖：

- 同一个 Session 内连续检视、修改和反馈不会创建新会话；
- 切换文档后保留通用上下文，清空旧候选稿，并将新文档作为源内容；
- 未应用候选稿切换文档时能够阻止静默丢失；
- 左侧基线保持只读，右侧候选稿可手动编辑；
- 接受、拒绝、手动编辑会正确更新候选稿和 Diff；
- 手动编辑后自检状态变为过期，且不会自动触发模型调用；
- 普通改稿和高影响改稿具有不同的确认流程；
- 当前差异、当前章节和全文反馈作用范围正确；
- 应用候选稿前正式文档不变化，应用后生成应用前历史版本；
- 另存为新文档后原文档保持不变，新文档成为当前文档并可继续使用同一 Session；
- 历史版本恢复经过候选稿和 Diff，不直接覆盖当前内容；
- 关闭并重新打开应用后可以恢复候选稿和改稿状态；
- 外部修改导致候选稿过期时，不发生自动合并或覆盖。

现有 DiffReviewPane、useDiffReview、选区操作、检视侧栏和文档修订能力可作为测试先例与底层测试接缝。

## Out of Scope

- 派生文档、文档分支和文档血缘管理；
- 多候选稿并行比较；
- 默认受保护内容或长期不可修改规则；
- 自动合并外部文档修改；
- 多 Agent 协作；
- 无限自主循环；
- 让 Agent 直接覆盖正式文档；
- 为每次 AI 行为创建独立对话；
- 事实核查、引用真实性验证等独立能力；
- 复杂的 Prompt 编辑器；
- 重新实现底层 Diff 算法和文档事务系统。

## Further Notes

本方案是一个 Human-in-the-loop Agentic Workflow：

- Agent 负责在候选稿中动态检视、修改和复核；
- Session 负责保持连续对话和当前工作状态；
- Diff 负责让修改可见；
- 用户负责最终接受；
- 文档系统负责正式写入、版本保存和恢复。

第一性原则是：不为少数边界场景引入新的文档对象和状态分支。所有新增能力优先落入“当前源文档、候选稿、Diff、历史版本”四个已有概念中。
