# AI 辅助研发工程实践调研：可用于 Typola 技术分享的行业基线

> 调研日期：2026-07-19
> 使用边界：只采纳工具厂商与工程组织发布的一手资料；本文提炼方法，不把厂商功能宣传等同于生产力结论。

## 一句话结论

AI 辅助研发不是“更会写提示词”，而是把传统研发中的**需求、上下文、实施、验证、评审、沉淀**改造成一个可审计的闭环：人决定目标、边界和是否合并；Agent 在受限上下文和权限内完成可验证的子任务；产物与失败经验回写为下一次可复用的项目资产。

## 1. 可直接引用的行业主张

| 研发问题 | 一手资料中的事实 | 可迁移的方法 |
| --- | --- | --- |
| Agent 为什么会“越做越偏” | Codex 在开工前读取 `AGENTS.md`，并按全局→项目→子目录建立有优先级的指令链；离当前目录更近的规则可覆盖上层规则。[OpenAI：AGENTS.md](https://learn.chatgpt.com/docs/agent-configuration/agents-md) | 把反复口述的技术栈、命令、禁止事项、文档义务和验收命令写进仓库；用目录级规则约束高风险模块。 |
| 上下文为什么是稀缺资源 | Claude Code 官方把上下文窗口称为最重要的资源，并建议将狭窄的探索交给子 Agent，避免主会话被日志和搜索结果淹没。[Anthropic：最佳实践](https://code.claude.com/docs/en/best-practices) | 主 Agent 只保留决策与交付；调研、定位、对抗评审等噪声大且相互独立的工作，委托给有明确输出格式的只读子 Agent。 |
| 何时适合并行 | Codex 官方说明：复杂且高度可并行的工作（例如代码库探索、多步骤特性计划）适合并行专用 Agent 后汇总；Claude Code 也强调子 Agent 的独立上下文、工具与权限。[OpenAI：Subagents](https://learn.chatgpt.com/docs/agent-configuration/subagents)；[Anthropic：Subagents](https://code.claude.com/docs/en/sub-agents) | 按“产物互不写同一文件、接口清楚、结果可独立验收”拆分；不要把相互依赖的实现硬并行。并行是减少等待，不是增加混乱。 |
| 为什么必须先定义验证 | Anthropic 明确建议为 Agent 提供测试、构建、脚本 diff 或截图等可读的通过/失败信号；没有验证，人的肉眼就成为唯一反馈环。[Anthropic：验证闭环](https://code.claude.com/docs/en/best-practices#give-claude-a-way-to-verify-its-work) | 每个任务在编码前写下验收命令或视觉基线；Agent 必须“改动→运行→阅读失败→迭代”，不能只报告“已完成”。 |
| Skills 如何避免重复发明轮子 | Codex 将 Skill 定义为可复用工作流；一个 Skill 由 `SKILL.md` 与可选脚本、引用资料、资产组成，并采用按需加载以节省上下文。[OpenAI：Build skills](https://learn.chatgpt.com/docs/build-skills) | Skill 不等于提示词收藏夹：把稳定的触发条件、步骤、检查项和脚本封装起来；先做少量高频高风险流程，如调研、测试、发布、评审。 |
| 规则与 Skill 如何分工 | GitHub 将简单、普遍适用的行为指令与复杂、按需的工作流区分为 custom instructions 与 agent skills；Codex 的技能同样是可按需加载的工作流包。[GitHub：Agent skills](https://docs.github.com/en/enterprise-cloud@latest/copilot/how-tos/copilot-on-github/customize-copilot/customize-cloud-agent/add-skills)；[OpenAI：Build skills](https://learn.chatgpt.com/docs/build-skills) | `AGENTS.md` 只保留始终成立的团队约束；调研、审查、发布、UI QA 等偶发多步骤流程才做 Skill。 |
| 怎样保存“AI 做过什么” | GitHub 的云端 Agent 工作在独立临时环境，可调研、计划、改分支并执行测试/linter；人再审阅 diff、迭代并决定是否创建 PR。[GitHub：Copilot cloud agent](https://docs.github.com/en/copilot/concepts/agents/cloud-agent/about-cloud-agent) | 将 AI 的可交付物固定为：计划、变更集、测试证据、评审意见和提交/PR；会话本身不是事实来源，Git 与文档才是。 |
| Agent 的安全红线 | OpenAI 将提示注入和私密数据泄露列为 Agent 风险：不可信文本不应直接驱动高权限行为；建议结构化输出、工具审批、输入防护和 trace/eval。[OpenAI：Agent safety](https://developers.openai.com/api/docs/guides/agent-builder-safety) | 网页、Issue、日志、外部文档都是不可信输入；敏感读写、联网、安装依赖、删除/发布应最小授权并要求人工确认；禁止把密钥或完整生产数据放进对话上下文。 |

## 2. 推荐的端到端作业法

这套“**SCOPE 闭环**”可作为分享的核心框架。它并非新流程，而是把成熟工程纪律接到 Agent 的长处上。

```text
S — Specify       明确用户价值、范围外项、约束、验收信号
 C — Contextualize 装载最小但完整的项目上下文与证据
  O — Operate      拆为低耦合、可回滚、可并行的可交付子任务
   P — Prove       用测试 / 构建 / 截图 / 人工验收证明，而非相信文字
    E — Evolve     将结论、坑、命令和规范沉淀为文档、规则、Skill、测试
```

### S — Specify：先把模糊愿望变成可检验的承诺

每个任务使用一页 Brief，至少含：

- 用户与场景：谁在何处遇到什么阻力；
- 成功条件：可观察行为与通过标准；
- 约束：兼容性、性能、隐私、不可触碰的模块；
- 范围外：这次明确不做什么；
- 验证：命令、测试用例、截图基线或人工检查人。

给 Agent 的好任务不是“做一个更好的编辑器”，而是“在当前 CM6 架构中为指定交互补齐 X；不新增依赖；`npm run typecheck` 与相关测试必须通过；提供前后截图/手测步骤”。

### C — Contextualize：交付“证据包”，不是粘贴整个仓库

按优先级提供：任务 Brief → 架构与模块入口 → 相关代码/测试 → 当前错误或截图 → 命令与规则。把长期稳定知识放进 `AGENTS.md`、`docs/ARCHITECTURE.md` 和模块 README；让 Agent 通过 `rg`、Git 历史和测试继续发现，而非在提示词里堆满文件内容。

上下文的质量可用三个问题检查：

1. 它知道什么不可改变？
2. 它能定位需要改的最小边界吗？
3. 它知道怎样证明自己完成了吗？

### O — Operate：以“可审阅变更集”为最小工作单元

推荐顺序：先只读探索并输出假设/风险 → 写短计划 → 实现最小垂直切片 → 运行验证 → 独立审阅 → 文档与变更记录。大任务可以并行调研、架构核查、竞品/官方文档核验和对抗评审；同一组件的写入由单一责任 Agent 串行完成，避免共享工作区冲突。

提示词里要显式写出操作边界：允许读什么、允许改什么、禁止触碰什么、何时停下请求决策。对外部副作用（安装、迁移、发版、删除、推送）默认改为“先展示计划与影响，再由人批准”。

### P — Prove：将“完成”换成证据链

建议按变更风险建立验证金字塔：

| 层级 | 典型信号 | 适用场景 |
| --- | --- | --- |
| 静态 | typecheck、lint、格式检查 | 所有代码改动 |
| 自动行为 | 单元/集成/E2E 测试、构建产物 | 逻辑、数据流、回归风险 |
| 视觉 | 截图对比、人工交互清单 | UI、编辑器、导出排版 |
| 对抗 | 独立 Agent/人工按反例审查 diff | 跨模块、高风险或看似“太顺”的改动 |

输出时把“运行了什么、结果是什么、尚未验证什么”与代码分开列出。失败不是可隐藏的噪音，而是下一轮输入。

### E — Evolve：把一次对话转化为系统能力

一次任务至少沉淀一个可复用资产：需求/设计决策、测试用例、排障记录、架构说明、变更日志或 Skill。优先沉淀那些“已经发生两次，且第三次再解释就浪费”的流程。定期删除过期规则；过长或互相矛盾的规则会降低而非提高 Agent 的可靠性。

## 3. 适合 Typola 的工具与能力栈

| 目的 | 建议能力 | 产物/控制点 |
| --- | --- | --- |
| 项目记忆 | `AGENTS.md` + 架构/产品/变更文档 | 规则分层、命令、边界、决策记录 |
| 快速定位 | `rg`、Git log/diff、只读子 Agent | 最小相关文件集和带证据结论 |
| 方案推演 | 短 Brief、架构审查、官方一手调研 | 选项、取舍、风险、验收 |
| 实现 | 任务分支或 worktree、最小 patch | 小而可回滚的 diff |
| 质量 | typecheck、测试、构建、浏览器/桌面截图 | 可复跑的验证证据 |
| 复核 | 独立对抗审查 Agent + 人工 PR 审阅 | 反例、回归项、安全项 |
| 知识复利 | Skill、模板、`lessons/`、变更文档 | 下次自动触发的流程能力 |
| 安全 | 最小权限、沙箱/隔离、审批、密钥管理、审计 | 受控副作用与可追溯记录 |

对于必须每次执行的动作，不能只依赖提示词记忆；应由 CI、pre-commit 或 Agent hook 等确定性机制来执行。这与 GitHub 对 hooks 可在 Agent 生命周期中运行命令的设计一致。[GitHub：Copilot hooks](https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/overview)

## 4. 常见反模式与替代动作

| 反模式 | 为什么失败 | 替代动作 |
| --- | --- | --- |
| 一句“帮我把它做好”后直接写代码 | 目标、边界和验收缺失，模型只能猜测 | 先产出 Brief 与验收信号，再开工 |
| 一次塞进整个仓库/大量日志 | 关键约束被稀释，上下文耗尽 | 先检索，再按任务组装最小证据包；噪声交给子 Agent |
| 让多个 Agent 同时改同一工作区 | 冲突、覆盖和责任不清 | 按文件所有权拆分；实现串行，调研/审查并行；必要时用 worktree |
| 只看 Agent 的“已完成”文字 | 文本不是运行结果 | 把通过命令、截图、测试输出作为交付物 |
| 让 AI 自行安装、联网、删除、发布 | 外部输入可注入，副作用难逆 | 最小权限、白名单、审批与可回滚操作 |
| 把每次经验都写成长提示词 | 规则过期、冲突且难发现 | 规则只留稳定约束；高频流程做 Skill；每项规则都附触发与验证方式 |

## 5. 分享中可使用的“成熟度”自检

| 级别 | 团队行为 | 下一步 |
| --- | --- | --- |
| L0：聊天式 | 用 AI 补全和问答，结果靠肉眼判断 | 为一个高频任务补测试/构建验收 |
| L1：任务式 | 能给清晰任务，AI 能提交小 diff | 建立项目规则与 Brief 模板 |
| L2：闭环式 | 每个改动有可复跑验证，失败会迭代 | 加入独立审查和视觉验证 |
| L3：系统式 | 规则、测试、文档、Skill 共同约束 Agent | 按风险分配权限与并行工作流 |
| L4：组织式 | 有度量、审计、可复用资产与持续改进 | 追踪交付周期、返工率、缺陷逃逸率，而非只看生成代码行数 |

## 6. 可直接带走的任务模板

```md
目标：<用户可感知的行为变化>
范围：只改 <文件/模块>；不改 <明确排除项>
约束：<技术、兼容、性能、安全、产品约束>
证据包：<入口、相关测试、设计/错误截图、官方资料>
实施：先只读定位并给出不超过 N 步计划；确认后做最小改动。
验收：运行 <命令>；完成 <手测/截图>；报告通过、失败与未验证项。
安全：不得读取/输出密钥；不得安装、删除、推送或访问生产环境，除非获得明确批准。
沉淀：更新 <CHANGELOG/架构/lessons/Skill> 中与本次变更直接相关的内容。
```

## 7. 对技术分享的落点

Typola 的案例价值不在于“AI 写了多少代码”，而在于把一个跨前端、Rust、编辑器交互、导出与桌面端的复杂项目，持续压缩成可验证的小闭环：先研究和建模，再做最小实现，以测试/构建/视觉检查收尾，并把规则、决策与踩坑沉淀回仓库。先进工具是加速器；真正可复制的护城河是**工程上下文、验证机制、边界控制和知识复利**。
