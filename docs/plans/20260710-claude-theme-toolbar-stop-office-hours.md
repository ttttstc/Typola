## Problem Statement

Typola 的「素笺」主题声称参考 Claude.ai 浅色，但当前配色是项目早期经验取值，与 claude.ai 实际浅色 UI 有明显漂移。同时选区浮条（SelectionFloatingBar）缺少"隐藏"入口，右键无法关闭浮条，并且 AI provider 卡死 / 返回异常时 Stop 按钮依赖 `onAgentExit`，前端无法强制收尾，会表现出 app 假死。4 件事都影响工作台的可信度，故一次性收敛。

## What Makes This Cool

主题对齐 Claude 是工作台"原生 AI 桌面"气质的一次校准；浮条隐藏独立按钮 + 右键关闭让浮条不再强制打断用户；Stop 按钮真无条件停让工作台在任何 provider 异常下都保持可操控感。

## Premises

1. 素笺对齐的是 **claude.ai 浅色 UI 配色**（米黄 ui / accent #d97757 / 蓝紫色链接），对齐范围是 canvas/paper/text/accent/ai-primary/success/danger/warning 主色，不含 markdown 内嵌代码的高亮色（那个跟 preview.css 解耦）。—— 用户确认 agree
2. 浮条隐藏入口是**两个独立按钮**（「本文档不再展示」 / 「全局隐藏」），不走「⋯」更多菜单；它们的位置在现有 polish/explain/review 右侧、⋯ 按钮之前。 —— 用户确认 agree
3. 右键关闭浮条是**全局上下文菜单触发**（`contextmenu` on document）；即使关闭的同时 EditorContextMenu 弹出也是可接受的，因为浮条本来就是"应该消失的悬浮物"。 —— 用户确认 agree
4. "停止按钮无条件停"由前端主动兜底实现：cancel 触发后立刻置 idle + 强制杀进程（后端已有 taskkill /T /F），不依赖 onAgentExit 汇入；同时 provider stream-handler 需要对非法 JSON / 非法 event 类型做 catch,避免 JS 层死循环。 —— 用户确认 agree

## Approaches Considered

### Approach A: 风格对齐 + UI 行为修正（本次选定）

Summary: 改 themes.css + themeRegistry.ts 把素笺细色全部对标 Claude.ai 浅色基准，浮条独立按钮 + onContextMenu 监听关浮条，useAgentSession.cancel 不 await exit 直接 idle 兜底。

Effort: M
Risk: Low
Repos:
- `scripts/theme-css-builder.mjs` 生成的 themes.css（需要改源头后 build:themes）
- SelectionFloatingBar/EditorPane 行为
- useAgentSession cancel 路径

Pros:
- 改动集中、原子化，可在单一分支交付
- 主题走现有 ThemeBuilder 管线，未来其他主题自动继承

Cons:
- 主题是 CSS 变量映射，改后必须跑 `npm run build:themes` + `tauri:build:local` 才能验
- 停止 1s 内 idle 需要前端兜底时放弃 exit 事件中的部分错误回传（需要可观测日志确保后端杀进程可被用户看到）

### Approach B: 仅对齐 canvas/paper + 仅修 stop

Summary: 只收敛 canvas/paper 两个基础色，浮条/右键留下个迭代。

Effort: S
Risk: Low
Pros: 风险最小
Cons:
- 主体深色 still drift,只调背景没解决作者真实诉求
- 浮条隐藏推迟会延长用户被浮条干扰的时间

### Approach C: 将 Clauid.ai 浅色抽成独立 theme slot

Summary: 新建 `claude-light` 主题保留素笺不动。

Effort: L
Risk: Low
Pros: 兼容原有偏好
Cons:
- 项目主题是 `plain-paper | night-current | ink-basin | abstract | brutalist` 五个位，再多一个意义不大
- 用户选择 "素笺已经对齐,只校准" vs "直接替素笺配色" 之间，用户选了直接替

## Recommended Approach

**Approach A** — 单分支合并交付，4 件事都是修边幅，不引入新的抽象，风险低。

## Open Questions

1. "停止 1s 内 idle" 的测量点: 从按钮点击到 Composer 输入框重新可输入的时间；前端兜底后 onAgentExit 晚到的不影响 idle 状态但要 console.warn 留观。
2. "全局隐藏" 仍走现有 `selectionFloatingBarEnabled=false` + 设置 UI 提示；"本文档" 走 session suppress（`dismissedForCurrentSelectionRef` 现在只按"选区"维度，需要升维到 path 维度并 session storage 持久化）。
3. 右键关闭浮条的开不是"全局都让 EditorContextMenu 不弹"，而是"同时弹也不影响",文档区右键关浮条同时允许 EditorContextMenu 走自己的 onContextMenu 判断。

## Success Criteria

- [ ] 素笺配色(canvas/paper/surface/text-primary/accent/ai-primary/success/danger/warning) 全部对比 Claude.ai 浅色基准,误差 ΔE<2
- [ ] 浮条出现后，两个独立按钮"本文档不再展示"和"全局隐藏"可见可点
- [ ] "全局隐藏" 落 `selectionFloatingBarEnabled=false` + 设"已全局隐藏" toast
- [ ] "本文档" session 内不再弹(切换文档恢复)
- [ ] 浮条在 `contextmenu` 事件触发后立刻 `visible=false`
- [ ] 停止按钮在 provider 卡死 / JSON 解析失败 / 流 hang 的任意情况下，**1s 内 Composer 回到 idle 可输入**
- [ ] 后端 taskkill /T /F 成功，进程句柄释放
- [ ] `npm run build:themes` 能够通过，`npm run tauri:build:local` 用户本地测试
- [ ] CHANGELOG 标注「素笺对齐 Claude.ai 浅色参考」

## Scope And Acceptance

- **问题定义**: 素笺配色对标 Claude.ai 浅色（主色 + AI 色，不含代码块高亮）；浮条加"本文档/全局隐藏"独立按钮；右键关闭浮条；停止无条件兜底
- **范围边界**: 仅改 `plain-paper` 主题配色；深海/墨池/抽象/粗野不动；仅改 SelectionFloatingBar，不动 SelectionAIMenu；cancel 仅修改前端等待逻辑，不新增后端接口
- **验收标准**: 见 Success Criteria 全部 checkbox + 用户本地 build 通过并演示
- **开放问题**: 停止按钮的错误回传是否 toast 化（后续迭代，本次不引入）
