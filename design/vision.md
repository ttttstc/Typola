# Typola · Product Vision

> 产品愿景与原则。v2.0 · 2026-04-19

---

## 1. 使命（Mission）

**把 Notion 级别的所见即所得体验搬到本地，让 AI 能直接读写你的文档。**

产品经理、运营、设计师已经开始用 AI 做需求分析、PRD、原型、用户手册，但卡点都在"文件在哪里"：飞书/Notion 的数据在云端，AI 够不着；Obsidian/Typora 在本地，但要求你懂 Markdown 语法，门槛劝退非程序员。

Typola 想做的是第三条路：**本地、标准 `.md`、Notion 级编辑体验**。文件就在你磁盘上，AI 工具可以直接读写，团队共享一个文件夹就能协作。

---

## 2. 一句话定位

**Typola 是一款面向 Windows 桌面的、AI 友好的本地 Markdown 编辑器，交互像 Notion，文件是标准 `.md`。**

- 打开即用，不需要配置，不需要账号
- 不需要记 Markdown 语法，像用 Notion 一样打字
- 文件以标准 `.md` 存在本地，AI 工具、VS Code、任何 Markdown 工具都能读
- 安装包 ≤ 30MB，启动快、占用低
- MIT 开源

---

## 3. 目标用户画像

### 3.1 主要用户（Primary）

**"用 AI 提效的产品团队成员"**
- 产品经理、运营、设计师
- 写需求文档、PRD、原型说明、用户手册、会议纪要
- 已经在用 Claude / ChatGPT / Cursor 等工具辅助工作
- 希望 AI 能直接读写自己的文档文件，而不是复制粘贴
- 不想记 `#`、`-`、`**` 这些符号

**"在意数据归属的个人写作者"**
- 不想让文档被任何云平台锁定
- 用过 Notion，喜欢其编辑体验，但担心迁移成本和平台风险
- 愿意用本地文件 + 自己的同步方案（iCloud / OneDrive / Git）

### 3.2 次要用户（Secondary）

**"需要本地协作的小团队"**
- 共享一个 NAS 或局域网文件夹
- 不想所有人都注册同一个云服务
- 对数据合规有要求（例如涉密、离线环境）

**"从 Typora 过来的轻度用户"**
- 只用 Typora 写日常文档，不依赖 KaTeX / 多格式导出
- 想要一个更现代、更"Notion 感"的替代

### 3.3 明确不服务的用户（Non-Goals）

- 不面向 **重度 Markdown 老手**（要 Front Matter / 脚注 / 自定义 CSS 的用户，v1 暂不支持）
- 不面向 **学术写作者**（v1 不做 KaTeX 数学公式，v2 再考虑）
- 不面向 **知识图谱用户**（不是 Obsidian 的替代，没有双链、图谱）
- 不面向 **实时协作团队**（v1 靠共享文件夹，不做 Google Docs 式协作）
- 不面向 **移动端用户**（v1 纯 Windows 桌面）

## 4. 核心价值主张

### 4.1 用户能得到什么（Value Proposition）

**"Notion 的编辑体验，`.md` 的数据归属，AI 的直接读写。"**

| 对比维度 | Notion / 飞书 | Obsidian / Typora | **Typola** |
|---|---|---|---|
| 所见即所得 | ✅ | ⚠️（偏源码） | ✅ |
| 需要学 Markdown 语法 | ❌ | ✅ | **❌** |
| 本地 `.md` 文件 | ❌ | ✅ | ✅ |
| AI 能直接读写文件 | ❌（云端） | ✅ | **✅** |
| 免费 / 开源 | ❌ | 部分 | **✅ MIT** |
| 配置成本 | 低 | 中-高 | **低** |
| 安装包体积 | N/A | 100MB+ | **≤ 30MB** |
| 厂商锁定风险 | 高 | 低 | **无** |

### 4.2 对 AI 工作流的价值

核心命题：**让 AI 成为你文档流程的一部分，而不是外挂。**

- PRD 在本地 `.md` → Claude Code / Cursor 直接打开并修改
- 会议纪要在本地 `.md` → AI 总结、提炼 action item 后写回原文件
- 用户手册在本地 `.md` → AI 根据代码更新文档，你只需 review

不需要 API、不需要插件、不需要导出导入。文件就在磁盘上。

## 5. 设计原则（Principles）

产品决策遇到分歧时，按以下优先级判断：

### 5.1 本地优先（Local-first）
文件 **永远**存在本地，以标准 `.md` 明文存储。任何功能都不应要求云账号。Typola 不提供云同步——同步交给 iCloud / OneDrive / Git / NAS。

### 5.2 AI 友好（AI-native by Storage）
Typola 不内置 AI 功能，但它的存在形式本身就是 AI 友好的：文件是标准 `.md`、路径是相对路径、图片放在 `.resources/`、没有专有格式。任何 AI 工具都能无障碍读写。

### 5.3 克制优于功能堆砌（Restraint over Feature Creep）
v1 故意砍掉很多"Markdown 老手"会想要的能力。这不是忘了做，是故意不做。每加一个功能都要问："非程序员会用到吗？会不会破坏轻量体验？"

### 5.4 所见即所得，零语法门槛（WYSIWYG, Zero Syntax）
用户不应该在界面上看到 `#`、`**`、`[]()` 这些符号。斜杠命令 + 选中浮动工具栏是主要交互路径，键盘快捷键是辅助。目标：让一个没听过 Markdown 的人打开就能写。

### 5.5 轻量是硬指标（Lightweight is a Hard Constraint）
安装包 ≤ 30MB，冷启动 < 1 秒，空闲内存 < 200MB。这不是"尽量"，是**硬门**。每次引入依赖都要过这关。为此我们选择 Tauri 而不是 Electron。

### 5.6 开源优于品牌（Open over Owned）
MIT 开源，无闭源商业版，无 Pro 订阅，无 VIP 功能。未来如果做托管服务（可选的云同步），不影响核心产品的免费开源属性。

### 5.7 纯黑白（Monochrome by Default）
UI 使用纯黑白，不引入品牌强调色。写作的注意力应该在内容上，不在软件本身。

## 6. 差异化（Differentiation）

### 6.1 相对于 Notion / 飞书
- **数据在本地**，不依赖云端，不被平台锁定
- **AI 可以直接读写文件**，不需要 API 或集成
- **安装包 30MB**，不是几百 MB 的客户端
- **开源 MIT**

### 6.2 相对于 Obsidian / Typora
- **无需学 Markdown 语法**，非程序员也能用
- 交互范式是 **Notion 式**（斜杠命令 + 块编辑），不是源码或半源码
- **更轻**（Obsidian 安装包 100MB+，Typora 闭源且已收费）
- 放弃插件生态以换取简单和稳定

### 6.3 相对于其他 Markdown 编辑器（Mark Text / Zettlr / MD Editor）
- 明确的**品牌与视觉语言**（Trio W 抽象标识、纯黑白）
- 明确的**工程质量标准**（类型完备、测试、文档、MIT）
- 对**中文用户**有原生友好度（字体、菜单、打包尺寸、Windows 优先）

## 7. 产品边界（Scope）

### 7.1 v1 覆盖（必须做）

**工作区与文件**
- 单工作区（选一个文件夹作为根目录）
- 左侧文件树（新建 / 重命名 / 删除 / 文件夹）
- 自动保存（停顿 500ms 后写盘）
- 外部修改检测（别的编辑器改了文件时提示用户）

**编辑体验**
- WYSIWYG 编辑区，不显示 Markdown 源码符号
- 斜杠命令（输入 `/` 弹出插入菜单）
- 选中文字浮动工具栏
- 右侧大纲（根据标题自动生成，点击跳转）

**内容类型**
- 标题（1-3 级）
- 粗体 / 斜体 / 删除线 / 行内代码
- 无序列表 / 有序列表 / 待办清单
- 表格
- 代码块（Shiki 语法高亮）
- 引用 / 分割线 / 链接
- 图片（粘贴或拖拽自动存入 `.resources/`，Markdown 里写相对路径）
- **Mermaid 图表**（双击块进入编辑模式）

**外观**
- 纯黑白亮色主题
- 纯黑白暗色主题
- （v1 不做用户主题系统）

**技术**
- Tauri（Rust 后端 + WebView2 前端）
- 安装包 ≤ 30MB
- 便携版 + 安装版

### 7.2 v2 考虑

KaTeX 数学公式、Front Matter、脚注、导出（PDF / docx / HTML / PNG）、用户主题系统、多标签、i18n、插件机制、自定义快捷键、Git 集成

### 7.3 永不做

- 富文本编辑（文字颜色、高亮、对齐、艺术字——标准 Markdown 表达不了的东西一律不做）
- 云账号 / 云同步（产品本身不做，用户可自行搭配 iCloud / OneDrive / Git）
- 商业锁定 / Pro 订阅
- 广告 / 数据收集 / 强制登录
- 知识图谱 / 双向链接（那是 Obsidian 的地盘）

## 9. 品牌叙事（Brand Story）

> 你的文档应该在你的电脑上，而不是别人的服务器上。
>
> 你的 AI 应该能直接读写你的文档，而不是让你复制粘贴。
>
> 你写文档的时候应该专注于内容，而不是语法符号。
>
> 这就是 Typola。三个 W 代表 **Win · Win · Win**——
> 你赢得数据归属，AI 赢得读写权限，开源社区赢得一个轻量开放的本地写作工具。

可用于 README 开头、官网 Hero、发布博文。

---

## 10. 长期愿景（3 年）

**做成"AI 时代本地 Markdown 写作的新默认"。**

- 在中文产品/运营社群里成为"本地 + AI 友好"这个细分场景的默认选择
- 在英文开源社区成为轻量 Markdown 编辑器里的首选（对标 MD Editor、Mark Text）
- 建立可持续的开源治理模式，不依赖单一作者持续投入
- 沉淀出一套基于 Milkdown/Tiptap 的**零语法 Markdown 编辑引擎**，可被其他本地优先应用复用
- 可选的托管服务（同步、团队空间）作为收入来源，但永远不影响核心产品的免费开源属性
