# Export Pandoc Roadmap

## 背景

当前导出能力优先覆盖零依赖的 `PDF` 和 `HTML`。`Word` / `Epub` 更适合走 `pandoc` 路线，但这条链路会引入额外二进制、模板、资源重写和跨平台打包问题，因此先记录方案，不在这一轮实现。

## 目标格式

- `docx`
- `epub`
- 预留 `odt` / `latex` 等后续扩展能力

## 建议架构

1. 渲染层仍然只负责发起导出请求和读取设置。
2. 主进程负责准备中间产物：
   - 当前文档 Markdown
   - 工作区资源目录
   - 导出临时目录
   - 可选的 `metadata.yaml`
3. 主进程调用 `pandoc` 二进制生成目标文件。
4. 导出完成后清理临时目录，并把错误信息回传到渲染层。

## 关键设计点

### 1. 二进制分发

- 优先评估随安装包分发 `pandoc`
- 需要按 `win/mac/linux`、`x64/arm64` 处理
- 二进制体积较大，可能需要拆分下载或按平台条件打包

### 2. 图片与资源

- Markdown 中的相对图片路径需要在导出前重写为 `pandoc` 可访问的实际路径
- `.resources/` 目录应整体复制到临时目录，避免原文档位置变化导致失效
- Mermaid 建议在导出前转换为 `SVG` 或 `PNG`

### 3. 样式映射

- `docx` 需要自定义 reference doc 模板，把标题、正文、引用、代码块映射到 Typola 的默认层级
- `epub` 需要内联一份更偏阅读器兼容的 CSS
- 代码高亮可优先退化为静态着色 HTML，再交给 `pandoc`

### 4. 可配置项

- Word 模板路径
- Epub 元数据（标题、作者、语言、封面）
- 是否嵌入图片
- 是否将 Mermaid / 数学公式预渲染为静态图

## 风险

- `pandoc` 体积和跨平台分发成本明显高于当前 PDF/HTML 导出
- Mermaid、复杂表格、任务列表在不同目标格式中的保真度有限
- `docx` 样式一致性需要额外模板维护

## 建议分期

### Phase 1

- 主进程集成 `pandoc` 调用链
- 打通 `docx` 最小导出能力
- 支持基础 Markdown、图片、表格

### Phase 2

- 加入 `epub`
- 支持导出模板和元数据
- 增强图片与 Mermaid 资源处理

### Phase 3

- 完善错误提示、缺失依赖诊断、打包策略
- 评估是否支持自定义导出 profile
