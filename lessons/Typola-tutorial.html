第 0 课

Typola 第一天:装好就能写

5 分钟。装好、打开一份文档、写一段字、保存。所有后面的课都建立在这节。

这 9 节课会带你看懂的 5 大能力

解决什么

写作者每天的 4 个真痛点:(1) 写一段话要润色,**但不想离开文档去 ChatGPT 复制粘贴**;(2) 想让 AI 帮生成报告 / PPT,但结果困在聊天记录里,不能落到文件;(3) 整篇改稿时怕 AI 一次覆盖丢原文;(4) 同事让你 review,你不知道意见给到哪段、最后怎么合并回正文。

Typola 怎么做

所有 AI 调用都在文档内或文档旁发生;AI 写的东西落到本地文件而不是云端 chat;改稿前先给你逐处对比再合并;检视意见是叠加的一层,导出前不污染原文。

跟别家不同

Typora 不带 AI;Notion AI 是云文档,产物在云上你看不见本地 Markdown;Typola 把"AI 是工序,文档是产物"这件事死守住 — 所以左栏是草稿(A 对话),中栏是成品(C 文档),从不把对话变档案库。

1. 装哪一版

Typola 目前只在 Windows 上发布。从 GitHub Releases 拿,两种包二选一:

安装版 Typola_*_x64-setup.exe

装在用户目录,自动关联 .md / .markdown / .html / .docx。日常使用选这个。 注意:Typola 也提供 .msi 格式;装到 Program Files 会弹管理员权限,大部分人不必用 msi。

免安装版 Typola_*_windows-x64_portable.zip

解压后双击 Typola.exe。适合临时测试、不想留安装痕迹、需要 U 盘带走的场景。 第一次启动会自动跑 WebView2 bootstrapper — 缺 WebView2 时会先装,无网就报错。

装好后第一次启动会做一次 WebView2 Runtime 预检。失败会弹窗告诉你去哪里装 — 跟着链接点过去装一次就好,以后不再弹。

快捷入门 4 步
  1. 装好 Typola
  2. 打开一份 .md 文档(后面第 3 步会教你方式)
  3. 默认就是阅读模式(所见即所得),不需要切
  4. 写完按 Ctrl + S 保存

2. AI 命令行要不要先装

看完第 0 节你就能写、就能保存。AI 那一大块要等本机有 Claude Code 或 OpenCode 之后才用得起来 — 后面的课都会用到。

如果你已经装了其中任一个,在 PowerShell 里跑一下确认:

claude --version
opencode --version

输出类似 claude 1.x.x 之类就行,装到具体哪个版本不重要。没装不影响第 0 课,但第 1 课开始就要至少一个能用了。

3. 打开你的第一份文档

四种打开方式,挑顺手的:

  1. 键盘 Ctrl + O → 选文件
  2. 拖拽:把 .md 文件从资源管理器拖到 Typola 窗口任意空白区
  3. 文件关联:装好 Typola 之后,双击 .md 默认就是它打开
  4. 命令行定位:Typola 启动时会读取最近的打开记录,直接出现在标题

几个常见文件类型都支持:

  • .md / .markdown / .mdx — Markdown
  • .html / .htm — 进入阅读/编辑链路
  • .docx — 只读预览(不要在 Typola 里改 Word,改完另存为 .md)
找不到合适的文档?

Typola 启动界面右侧的"快速打开"(Ctrl + Shift + P)会列出最近打开过的文件和当前常用工作目录,任选一个往下做。

4. 写一段再保存

默认就是阅读模式(WYSIWYG),你看到什么就是最终阅读的样子。可以直接:

  • 输入一段普通文字
  • 用 # 起标题、- 起列表、``` 起代码块 — 输完空格就立刻渲染
  • 图片可以直接粘贴剪贴板截图,Typola 会在文档同目录的 assets/ 下保存一份,并自动插入引用

第一次写完一定要按 Ctrl + S 保存。标签栏的文件名旁边会出现小圆点表示脏改,保存后小圆点消失。

Ctrl + Shift + S
另存为(可换目录、改后缀)
Ctrl + F

Ctrl + H

关于"自动保存"

Typola 默认关闭自动保存(这是经过考虑的选择,不是 bug)。要打开去 设置;但前几课建议就用 Ctrl + S 手动保存,养成"改一段就存一段"的肌肉记忆,后面跟 AI 协作时才不会因为意外覆盖丢原稿。

5. 认识三种文档模式

Typola 同一篇文档,可以在三种模式之间切换 — 顶部那条三段控件就是切换器:

阅读模式(默认,WYSIWYG)

边写边看到接近终稿的排版。适合 90% 的写作时间。第一周你只用这一种就好。

心流模式

左侧栏让位给 AI 工作台;你可以一边编辑一边让 AI 帮忙。第 2 课展开。

检视模式

进入"给文章提意见"的节奏,不在原文写脏字,而是叠加一层意见;最后导成 review.md 给协作者看,或一键让 AI 按意见改稿。第 5 课展开。

三者的键盘切换:

Shift + A
心流 ↔ 阅读来回切
顶栏三段控件
任意两个模式之间点选

❌ 第 3 步图片没出现 / 文件没保存 / 找不到 assets/:去 陷阱区对照一下。

陷阱

  • 文件是 .docx 直接在 Typola 里改 — Typola 把它当只读预览,改了等于丢。开 Ctrl + Shift + S 另存为 .md,改 .md 才安全。
  • "自动保存开着怎么还是不存" — 默认就是关的。先去 设置 里勾上。
  • Ctrl + S 没反应 — 文档是从 Typola 之外"刚刚拖进来"的状态,会被当成未命名 — 看标题栏有没有 * 或文件名是不是 未命名。
  • WebView2 启动报错 — 跟着弹窗提示去 microsoft.com/webview2 装;装完重启 Typola。

能力定位

解决什么

"这段写得不好,但我懒得去 ChatGPT 复制粘贴" — 写一段就被打断一次的痛。 Typola 把 AI 改稿压成一条浮在选区上方的横排工具条,不离文档、不开新窗口。

Typola 怎么做

选中文字后延迟 160ms 自动出现的 SelectionFloatingBar。点动作 → 起一个隐藏的 AI 会话(只为了拿替换文本)→ 把"原文 vs 新版"叠回选区下方一张结果卡 → 你点 采纳替换 才真的写回正文。

跟别家不同

VS Code Copilot 是字符级补全(改你还没写的字);Notion AI 是整页 AI 改稿(一键覆盖,不可拒)。 Typola 走"选段级 + 原地对比 + 逐处采纳"比 Copilot 改得多,比 Notion 改得可控。 右键菜单 Ctrl + K 还有 4 个动作(缩写/扩写/校对/自定义)不在浮条里 — 浮条只挑了最常用的 3 个。

前置

  • Typola 已装,且至少一个 claude / opencode 可用
  • 设置 → AI 执行 里能看到 CLI 检测通过(版本号不是空的)
  • 编辑器里有一份写了一半的文档(自己随便写一段就好,不需要写得好)
CLI 没检测到?

常见原因:PowerShell 的 ExecutionPolicy 挡了 npm 的 .ps1,或装到了用户目录但 PATH 里没有。 最简单的兜底:在 设置 → AI 执行 → 可执行路径 里手动指定完整路径(例如 C:\Users\你\.npm-global\claude.cmd)。 改完点一下"检测",看到版本号就 OK。

1. 浮条润色 — 3 步让 AI 改完并采纳

这是最常用的动作,适合大部分"这段能更好"的场景。

步骤 1
在文档里选中一段中文/英文文本(鼠标拖选 3-5 句话,空段落、整段都行)
步骤 2
松开鼠标 ~160ms 后,选区上方会出现一条小工具条:「润色 / 解释术语 / 加检视意见」
步骤 3
点「润色」→ 选区下方贴出一张对比卡「原文 vs 新版本」+ 采纳替换 / 取消

点 采纳替换:原文被精确替换,自动加进撤销栈(Ctrl + Z 能撤回)。 点 取消:什么都没发生,文本还是你的。

采纳的底线逻辑

浮条不偷你文字。原文永远在磁盘的当前文档和撤销栈里 — 哪怕点了「采纳替换」,Ctrl + Z 也能一步步回退。 对比卡是叠加的,采纳前你看得到「会变成什么」。

2. 浮条只有 3 个动作?右键还有 4 个

浮条故意只挑了最高频的 3 个(润色 / 解释术语 / 加检视意见)。 其余的 4 个 — 缩写、扩写、校对、自定义 — 都在 右键菜单 / Ctrl + K 里:

  1. 选中一段文字
  2. 按 Ctrl + K 或右键 → AI 动作
  3. 在列表里选「缩写 / 扩写 / 校对 / 自定义」
  4. 弹出的输入卡里可以填具体要求(如"改得更口语","目标长度 80 字"),Ctrl + Enter 发送
  5. 同样回一张「原文 vs 新版本」对比卡,采纳替换或取消
什么时候用哪个?

润色 — 文字通顺但不够精炼,要打磨。
缩写 — 长了,要砍一半。
扩写 — 太干,要补论据/例子。
校对 — 怀疑有错别字/语法/标点问题,只想扫一遍不改写。
解释术语 — 选了一个词/概念想加一段人话注解(不替换,只展示),你可以复制走。
自定义 — 上面的都不合适,自己写要求(例如"翻成英文,保留技术名词")。

3. 解释术语 — 这是唯一一个"不替换"的动作

「解释术语」的特殊之处在于:结果不打算替换你选中的词,只是解释一下给你看。 所以它的结果卡显示成单文本块 + 复制按钮,没有"采纳替换":

典型用法:文档里有个专业术语,你写作时需要看一眼定义但不想让文档里多出这段解释。 点解释 → 复制走 → 关卡 → 文档不变。

浮条只有 3 个 — 加检视意见要等第 5 课

浮条第 3 个按钮「加检视意见」是给检视模式铺路的:它不会调 AI,只是给选中段落记一条意见。 真正发 AI 改稿的入口在 右侧检视面板,要切到检视模式才出现。第 5 课再展开。 这一课先只用「润色 + 解释术语」。

试一试

10 分钟任务:让 AI 改 3 段
  1. 在自己的笔记文档里选一段长句子(超过 50 字),点浮条「润色」,看新版本,采纳。
  2. 再选一段,按 Ctrl + K 在右键菜单里选「缩写」,写要求"砍到一半长度",采纳。
  3. 选一个你不熟的术语,点浮条「解释术语」,看结果,点「复制」,粘贴到文档里随手位置,再撤销粘贴(练一下 Ctrl + Z)。
  4. 最后按 Ctrl + Z 几次回到最原始的版本 — 确认所有 AI 改动都能回退。

✅ 4 步全过 = 进入第 2 课:AI 工作台开箱。

陷阱

  • 浮条不出现 — 最常见是文档在"源码模式"(不是阅读模式),或你刚松开鼠标在拖选过程中。 等半秒别动鼠标,或在阅读模式(WYSIWYG)里选。
  • 浮条出现但点不动按钮 — 鼠标落在了按钮之间的空隙上。点按钮中央的图标或文字。
  • 点了「润色」卡在 loading 转圈 — CLI 进程没起来,或网络/鉴权问题。 等 8 秒还没结果会自动跳到错误卡,点「重试」,再不行查 设置 → AI 执行 的检测。
  • 采纳替换后文本看起来没变 — 通常是 AI 的"新版本"几乎等同原文(就是只换了几个字)。 退回看对比卡前后对比。
  • Ctrl+Z 不能回退 AI 改动 — 在阅读模式下是支持的。如果不行,先切到阅读模式再试。

能力定位

解决什么

整篇稿子写一半,你想问 AI"这一段论据够不够"、"开头能不能更吸引人"、"有没有逻辑漏洞" — 这些不能靠浮条的选段动作解决,因为你要把上下文作为整体拿出去。需要一个能围绕当前文档来回对话的工作区。

Typola 怎么做

左侧栏在"文件树"和"AI 工作台"之间切换(不并存)— 选了 AI 工作台,文件树收起让位,编辑器保持全高不被挤压。 Typola 用本机 CLI 跑 headless 流式协议(Claude Code 走 stream-json,OpenCode 走 JSON),所有工具调用、思考、正文、应用内卡片化展示 — 你不用开终端、不用手动回车。

跟别家不同

Claude Desktop / ChatGPT / Cursor 是"对话为中心",文档是附加品;Typola 反过来 — 文档是 C 位,对话是草稿。 所以 Typola 不做"会话存档库",不做"AI 用量统计",不做"跨会话搜索";只做"围绕当前稿子 + 当前文档附件"的少量并行任务。这也是为什么用 Pill 而不是 Tab 管对话(下节课展开)。

前置

  • 你已装好 claude 或 opencode,Typola 设置 → AI 执行 检测通过
  • 已经完成第 1 课(浮条能正常工作)
  • 至少有一份写了一半的文档在打开

1. 唤出 AI 工作台

两种方式 — 任选:

Shift + A
心流模式 ↔ 阅读模式来回切
顶栏三段控件
手动点"心流模式"

唤起后你会看到三件事同时发生:

  1. 左侧栏:文件树收起,弹出 AI 工作台(对话流 + 输入框)
  2. 编辑器:高度不变,继续全高展示当前文档(不被对话框挤)
  3. 右栏:SkillHub 出现(第 4 课展开)

再按一次 Shift + A 或点回"阅读模式",就回到只有文件树 + 编辑器的常态。

"心流模式"这个名字的意思

心流 = flow = 沉浸写作状态。Typola 不想让你被对话框的常驻存在打断,所以左栏只在需要时让出来; 它跟"AI 工作台"是同一区域,只是更强调"这是写作者的沉浸写作位"。 看到"心流模式"四个字,脑子里就是"左栏切到 AI 工作台"。

2. 发第一条消息

左栏底部输入框(Composer)就是入口。

  1. 在输入框写一句问题(例如「帮我找出这篇文章里论据最弱的 3 段」)
  2. (可选)勾上你想让 AI 看见的上下文芯片:当前文档 默认就在
  3. 点 发送 或 Ctrl + Enter
  4. 等 AI 流式输出 — 看到下方出现卡片就开始收到回应

发送后输入框下方会出现一条取消按钮,任何时候想中止 AI 当前输出就点它,等同于"放下笔"。

3. 读懂 5 种流式卡片

AI 输出不是一坨文本滚下来,而是按内容类型分卡片出现 — 你得能识别:

卡片 它是什么 你该怎么做
思考卡(Thought) AI 在真正打字前的内部推理。默认折叠,点开看灰色的文字流。 通常不用读,遇到 AI 输出诡异再看。
正文卡(Markdown) AI 给你的回答本身(打字机式逐字出现)。 读它。打字机滚动时还能左右滑或停止发送。
工具卡(Tool) AI 调用了 Read / Edit / Bash / Glob 之类工具。低关注度工具默认折叠。 折叠着就好;AI 在读你的文档是正常的。
Done 状态栏 一轮结束时的耗时 · 输出 token · 估算花费。 看一眼知道这轮成本。
错误诊断卡 AI 卡死 / 鉴权失败 / 模型无权限 — Typola 把 CLI 的退出码翻译成人话。 看诊断文本 → Retry 或按它给的指引修。
怎么识别"AI 现在还在想" vs "AI 已经放弃"

正常等待 → 流式卡片在追加(打字机还在滚)。 诊断卡 出现 → AI 已经停了,在告诉你为什么失败。 空白超过 8 秒 → 多半 CLI 卡死,可以按取消,再 Retry。

4. 切换 Provider(Claude Code ↔ OpenCode)

Typola 支持两个 provider,Composer 底部有一个小药丸按钮组(不是设置里的): 点 Claude 就切 Claude Code,点 OpenCode 就切 OpenCode。

切 Provider = 新会话

这是重要规则:切换 Provider 会弹确认「会开始新对话,旧会话上下文不会跨 provider 延续」。 为什么:Claude 的内部状态格式跟 OpenCode 不一样,跨 provider 续传只会让 AI 一脸懵。 想保留上下文就别切 Provider;想试不同模型,在同一个 Provider 下用它的 CLI 自带切换机制(例如 Claude Code 的 --model 参数)。

切完 Provider 之后,输入框、Done 栏、错误诊断卡的视觉风格也会跟着换 — Claude 用它原生 stream-json,OpenCode 用 JSON 协议,UI 形态相同。

5. 多轮对话 — 上下文延续

发第一轮后,继续在输入框打字再发第二轮,**Claude 会记得你上一轮说了什么**(它的 session UUID 由 Typola 内存维护)。

这是写作者最常用的模式 — 同一稿子来回迭代:

👤 : 我写了篇关于 X 的稿子,帮我看开头能不能更吸引人
🤖 : (输出建议)
👤 : 那段能不能换成更克制的语气?
🤖 : (在上一轮建议的基础上改)
👤 : 改完了,我整篇都在这,最后整体顺一遍
🤖 : (通读)
多轮的边界

同一运行期内:会话上下文保留(每轮发请求时 Typola 把 session UUID 给 CLI 续传)。
重启 Typola:会话丢失(内存 Map 不持久化)。后者是已知限制,跨重启的持久化在路线图里(M3+)。

第 3 课

上下文芯片:让 AI 看见你要的素材

8 分钟。把当前文档、附件、参考资料、MCP、Plugin 拼成 AI 能看见的 prompt 上下文。

能力定位

解决什么

让 AI 改稿、问问题,光凭当前文档不够 — 你要把参考资料、调研稿、风格指南、上周的周报一并喂给 AI; 或者你想让 AI 知道某个 MCP 工具能用、某个 Plugin 目录里有哪些 skill。 这种"把哪些东西带入这一轮对话"的精细控制,是普通 chatbox 做不到的。

Typola 怎么做

Composer 上方有一条上下文芯片(ContextChipStrip)— 每行是一个 chip:📄 当前文档、📎 附件 1、📎 附件 2、🔌 MCP 文件、🧩 Plugin 目录。 每个 chip 可以单独去除(比如不要当前文档),也可以通过 Composer + 菜单加新附件。 这些 chip 会转成 CLI 的路径参数 + prompt 文本喂给 AI;AI 真正去看这些文件再回答。

跟别家不同

Cursor / Copilot 把"项目索引"全局化(扫整棵树给你做检索),但写作者不需要; Notion AI 把"上下文"做成页面链接注入,但写作者的素材不在 Notion 里。 Typola 走"显式芯片",你每轮要什么给什么 — 不偷偷自动加文件、不做大文档 RAG 索引。 想让 AI 看什么,就加 chip;想去掉当前文档,直接点 chip 上的 ✕。

前置

  • 已完成第 2 课(AI 工作台能开能发消息)
  • 有一份打开的文档
  • (可选)准备一份 markdown 或 txt 的"参考资料"在本地 — 比如风格指南、调研稿、上周周报

1. 认识芯片行

唤出 AI 工作台(Shift + A)后,看 Composer 输入框上方那条横向小标签:

[📄 当前文档 note.md] [📎 style-guide.md] [+ 添加附件] [+ .mcp.json] [+ Plugin]

每个 chip 是一行精炼信息:图标 + 文件名 + 移除 ✕。 这条线就是这一轮 AI能看见的全部文件清单。默认情况下:

  • 当前文档 — 始终在场(除非你主动点 ✕)
  • 附件 — 你按需加,无默认
  • MCP 文件 — Composer + 菜单开,默认无
  • Plugin 目录 — 设置 里配置后才出现
芯片的真实作用

每个 chip 都会被 Typola 转成 CLI 调用参数:

  • Claude Code:文件路径作 --add-dir + prompt 文本里写「参考以下文件」
  • OpenCode:每次发送都重复 --file <path>,prompt 文本也带参考文件 block

AI 收到 prompt 后会调用 Read/Edit 工具去读这些文件;不是 Typola 提前把整文件内容塞 prompt。

2. 加附件文件

  1. 点 Composer 左下角的 + 按钮(在输入框外/底部,不是 chip 上的)
  2. 弹菜单里选「附件文件」
  3. 文件选择对话框打开,选一个或多个 markdown / txt / html
  4. 选择完毕,Composer 上方多出 📎 文件名 chip

加完即用。下一次发送时 AI 就能读它。

用附件的常见 3 个场景
  • 风格指南:让 AI 改稿时按指南(术语用词 / 句式长短 / 标题层级)
  • 参考资料:让 AI 把对话里"我引用某论文"的那种话术填到引用文件的具体出处
  • 上周周报:让 AI 写本周周报时,自动知道上周期项目进展、人员分配

3. 去掉某个芯片

暂时不想要某个 chip 的引用?点 chip 上的 ✕ 就消失:

  • 点当前文档的 ✕ → 这一轮 AI 看不到你这篇稿子(只看你直接发的 prompt 内容)
  • 点附件的 ✕ → 这一轮不再喂这份附件(文件还在磁盘上,下次想用再加回来即可)

常见场景:你在写一段说明 A 的小段,AI 没必要读整篇稿子。这一轮就把当前文档 ✕ 掉。

关掉当前文档 ≠ 重置全局

很多人担心「关掉当前文档后,下一轮是不是还得加回来」。 不是的:你切回阅读模式之前,关 chip 的状态保留;切回阅读模式 / 退出心流模式也保留。 真正清芯片的是切文件(tab) — 切到下一篇文档时,旧芯片整批清空,新文档的当前文档 chip 顶上。

4. 加 MCP 配置(MCP 文件)

如果你的工作目录里有 .mcp.json(Claude Code 或一些 MCP 配置格式),想在这一轮让 AI 用 MCP 工具,不用单独配置,只要在 Composer 的 + 菜单选「.mcp.json」就加进去。

它会以 chip 的形式出现,告诉 CLI 这一轮要用哪份 MCP 配置(然后 CLI 自己解析)。

MCP 配置的常见位置

用户级:~/.claude.json 或 ~/.opencode/mcp.json
项目级:当前工作目录的 .mcp.json(Composer + 菜单的工作目录 = 你当前 Composer 上看到的 cwd)

5. 加 Plugin 目录

Claude Code 的 --plugin-dir 参数让你告诉 CLI 你的 skill / 命令面板加载哪个目录。 Typola 在 Composer + 菜单里提供「Plugin 目录」— 选择后这路径每次发请求都会被附加为 --plugin-dir <path>。

  1. Composer + 菜单 → 「Plugin 目录」
  2. 选一个包含 commands/ 或 skills/ 的目录
  3. 之后发请求 AI 就能调那个目录里的命令
Plugin 目录 = Skill 目录?

接近。Typola 内部有一个固定的 SkillHub 让你用卡片选(第 4 课展开),那走系统推荐。 Plugin 目录 是更底层的"告诉 CLI 这个路径下有我的东西" — 适合临时加一个本地 skill 仓库。

6. 切文档 = 整体重置芯片

写作者的常态:边写稿边开 AI 工作台,但 AI 工作区不一定只用一篇稿子。 Typola 的设计:

动作 芯片会怎么变
编辑器里切换文档 tab 旧文档的当前文档 chip 被新文档的当前文档 chip 顶替;附件芯片全部保留(它们是给当前 session 用的,跨文档仍生效)
切 Provider(Claude ↔ OpenCode) 芯片清空,新 Provider 会话重新建立
开新会话(Pill 第 6 课展开) 芯片按新会话需要重新加;不会从旧会话迁移

需要准备:一份风格指南(或随便 5 条用词规则 + 一句话品牌语)+ 一份调研稿(引用 + 数据)。

  1. 在工作目录下创建 style-guide.md(自己随便写 5 条规范,例如「用『用户』不用『客户』」、「段首不空两格」)。
  2. 创建 research-notes.md,塞进 3 条"摘自某地的引用"。
  3. 打开你的正稿,Composer + → 附件文件 → 选这两份。
  4. Prompt 写:「请用 style-guide.md 的规范改我的稿子,需要数据引用时从 research-notes.md 里取」。
  5. 看 AI 输出是否符合 style-guide 规范、是否用了 research-notes 里的内容。

陷阱

  • 加了附件 AI 说"看不到" — 多半是路径不对(Typola 传的是绝对路径)。 检查 Composer + 时选的文件是不是你预期的(Composer 不一定跟编辑器同目录)。
  • AI 不读 MCP 文件 — MCP 是你自己的 .mcp.json 格式要正确;Typola 只传路径,不在前端预先解析。
  • Plugin 目录加了没生效 — 重启 Typola 或重新发一轮,CLI 偶尔需要新一轮才挂新 Plugin。
  • 切文档后附件还在 — 这是设计,不是 bug。芯片生命周期不等于文档生命周期;你想清除就手动 ✕。
  • + 菜单里看不到 MCP 文件 — 因为当前 Composer 工作目录还没确定(Picker 还没选)。先在 Composer 顶部 WorkingDir Picker 里选一个工作目录。

第 4 课

Skill 场景:报告 / PPT / HTML 的固定套路

10 分钟。把"调 skill"从手敲命令变成点选 — 三个场景、十几个预置。

能力定位

解决什么

你装了 claude --version 能跑、能调 skill,但手敲 /huawei-style-ppt-skill 这种命令很别扭 — 还得记每个 skill 的名字、出处、能干啥。 Typola 把本机装的所有 skill 按"你会用它做什么任务"分类,你只管选场景、选 skill。

Typola 怎么做

右栏 SkillHub 是一组"场景卡"(首批三个场景:报告生成 / PPT 制作 / HTML 制作)。 每个场景下挂着系统推荐 skill + 你手动加进来的 skill。 点已安装的 skill → 开一个独立会话(标题= skill 名,输入框预填 /<skill-name> ,独立 session UUID)。 每个 skill 一个独立上下文 — 这是 Typola 的护城河。

跟别家不同

Cursor / Claude Desktop 让你手敲 / 调命令(命令面板),Coze / Dify 给你通用 marketplace。 Typola 不做 marketplace(那是产品错位),也不做"全量本机 skill 列表"展示 — 它只按场景把高频用法浮上来, 想看全部或加新 skill 走 + 添加 skill 弹窗。本机 skill 不是 Typola 的资产,是 Claude/OpenCode 的 — Typola 只"引用",不"拥有"。

前置

  • 已完成第 2、3 课
  • 本机至少装了 claude 或 opencode
  • (可选)~/.claude/skills/ 下有 1-2 个 skill 文件夹(没装也没关系,会演示"未装时让 Claude 装"的流程)

1. 认识右栏 SkillHub

唤出心流模式后(Shift + A),看右侧栏。 上半段是 SkillHub(场景卡),下半段会留给产物中心(第 7 课展开)。

SkillHub 默认是一级视图:

┌─ SkillHub ──────────────────┐
│  报告生成     PPT 制作      │
│  HTML 制作                   │
└──────────────────────────────┘

3 张场景卡(报告生成 / PPT 制作 / HTML 制作)。系统内置首批。

为什么只有 3 个场景?

首批覆盖写作者最常要 AI 干的"出物"任务。其它(写周报 / 写 PRD / 写书评)暂时归到"报告生成"里扩展 — 不要让"全量场景"代替你思考,场景卡是预置的、克制的。

2. 三个场景分别预置了什么

场景 预置 skill(本机已装时显示卡片,未装置灰)
报告生成 首版空模板 + 你手动加进本场景的 skill。第 0 版故意空着,等你 "用出来" 才好推荐。
PPT 制作 huawei-style-ppt-skill、guizang-ppt-skill、huashu-slides、baoyu-slide-deck
HTML 制作 frontend-slides

3. 点已装 skill 开新会话

点一下场景卡(例如「PPT 制作」)→ 进入二级视图,看到 PPT 场景里挂的 4 张 skill 卡。 点已装的那张(卡片非灰色)→ 发生 4 件事:

  1. 左栏 新会话建立(标题等于 skill 名,例如「huawei-style-ppt-skill」)
  2. 输入框预填 /huawei-style-ppt-skill (光标在末尾,你打 prompt)
  3. 独立 session UUID生成 — 这个 skill 的多轮上下文跟其他会话完全隔离
  4. Composer 底部出现 Provider pill(默认沿用当前 Provider,Claude → Claude)

直接在预填后面写任务:

/huawei-style-ppt-skill 基于我的本周总结生成一份 8 页汇报 PPT

发出去,AI 加载 skill 并按其内部流程生成 PPT(产物会落 .typola-output/<会话>/,第 7 课展开)。

为什么"每个 skill 一个独立会话"?

护城河。写过周报 + 改文档的人都有过这种惨痛: 用同一个 AI 会话跨任务,AI 把上一次的"项目背景"擅自带了进来,导致周报开头多了"我们"这种 没必要的语境。 Typola 强制每个 skill 一个新 session UUID,从根上切断泄漏 — 你的周报 skill、PPT skill、技术笔记 skill 永远不会串味。

4. 未装 skill → 让 Claude 装

PPT 场景里有 4 张预置卡,但你本机可能一个都没装。 这时候卡片置灰但仍然保留 — 你能看到"它能干什么"(系统推荐摘要),从而判断要不要装。

点未装的卡片:

  1. Typola 不直接动文件系统(设计红线 — Typola 不替你安装,以免装错/装到错位置)
  2. 而是把一条安装指令 prompt 发到当前 AI 工作台会话
  3. 指令格式大约是:请帮我安装 Claude skill:huawei-style-ppt-skill。用途:<系统推荐摘要>。来源:<系统配置里的安装路径,或"未提供来源">。安装后请确认该 skill 可以通过 /huawei-style-ppt-skill 调用。
  4. AI 用 CLI 原生的能力去装它(比如 mkdir + 克隆 git repo + 写 SKILL.md)
  5. 装完你回 SkillHub 刷一下(右上角 ↻),卡片就显示"已装",可以用了
为什么不让 Typola 直接装?

原因 1:安全 — 让第三方编辑器写用户目录外的文件系统 = 不可控的攻击面。 原因 2:溯源 — 你从对话里能看到 AI 是用哪条指令装的,以后想自己改路径也不用"猜 Typola 怎么装的"。 让 AI 用它原生能力干活,Typola 只"派活",这是「我司不替你装软件」的克制。

5. 添加本机已装的 skill 到场景

你可能自己装了几个 skill 不在系统推荐里,想加到「报告生成」或「HTML 制作」场景:
二级视图右上角有 + 添加 skill 按钮。

  1. 点「+ 添加 skill」
  2. 弹窗列出当前 CLI 本机已经认识的全部 skill(不只是 Typola 系统推荐的)
    — 你可以搜索关键字过滤
  3. 勾选你想要的那条 → 确认
  4. 返回该场景二级视图,看到多了你刚加的卡片(图标 可识别为"用户加"标签)
  5. 系统预置不可删(避免误删),用户加的可删(小卡片上有 ✕)
"添加"和"安装"是两回事
  • 安装 = 文件系统上多了一个 skill 目录(Typola 不做这个)
  • 添加 = 本机已有 skill,告诉 Typola "我想在报告生成这个场景里看到它"(只是 skill-hub.json 多一条用户配置)

Typola 的 skill-hub.json v2 只存用户数据,绝不复制 skill 内容进系统模板。 升级 Typola 也不会丢你的用户配置。

6. Claude 与 OpenCode 在 SkillHub 上有什么不同

同一个 SkillHub 卡片在两个 Provider 下默认都显示,但实际调用方式完全不同:

Provider skill 怎么调用 扫描什么
Claude Code prompt 里写 /skill-name ... 给 CLI,触发原生 slash 命令 ~/.claude/skills/ 下的所有 skill
OpenCode CLI 参数 --command <name>;prompt 文本作为 opencode run 的位置参数追加 ~/.opencode/commands/ + ~/.opencode/command/ + opencode.jsonc 里的 command 条目
"装了但两个 Provider 都看不到" — Provider 隔离读得懂

Claude 的 skill 和 OpenCode 的 command 不互通 — 它们存在不同的目录,定义格式也不一样。 你装了 Claude 的 ~/.claude/skills/foo,那它只在 Claude Provider 下能用; OpenCode 看到的是 ~/.opencode/commands/foo.md。 想两边都能用,各装各的。

  1. 弹出检视意见编辑器浮卡(贴选区下方)

页面变化:

  • 被批注的段落整段高亮(主题色淡背景,不是 AR 涂红)

Typola 不用 emoji + 不用自定义数据属性,直接用 > **检视意见,请处理**:<意见> 这种 md blockquote 形式 — 这样导出的 review.md 本身就是干净的 Markdown,人类可读、AI 可懂。 等于一份"喂给 AI 的 prompt 格式" + 一份"给同事看的可读版"。

6. 内联审阅态 — Y / N / ⏎ 鼠标级键盘

这是 Typola 真功夫。中间编辑器不是右栏铺开,而是直接接管。你能看到:

  • 顶部条「审阅 AI 改稿 · 第 N/总数 处 ▲▼ 全部采纳 / 全部拒绝 / 应用 ✕」
  • 左侧(占主要区域):分段后的混合视图
    → unchanged 段:直接渲染
    → modified 段:原文删除线 + 新版高亮 + 该段下方贴 ✓/✗ 按钮
    → added 段:绿色高亮 + ✓/✗
    → removed 段:删除线 + ✓(采纳删除) / ✗(保留)
  • 右侧滚动条:改动地图 — 每处一个小标记,当前处放大高亮,点就跳

键盘流(进入审阅后全程不用鼠标):

Y
采纳当前 hunk,自动跳到下一处
N
拒绝当前 hunk,自动跳到下一处
Enter / ↓
跳到下一处
Shift + Enter / ↑
跳到上一处
Esc
退出审阅态(若你动过 hunk 会有轻确认 "放弃本次 AI 审阅?")
每处默认状态 = 采纳

这是产品决策:用户最常的意图是"AI 改得还行就用",所以每处 diff 默认就是接受状态。 用户只需对不想要的改点 ✗,而不是对每处都去点击 "采纳"。 想反向操作(默认拒绝)就开 全部拒绝。

7. 点「应用」合并回原文

所有处过完一遍,点「应用」→ Typola 把当前各处状态合并成完整文本,调编辑器 setValue 写回当前文档 → 退出审阅态 → 文档此刻变脏(因为它已经被改了)。

写回是一次原子操作:一次 Ctrl + Z 能完整回到改稿前的原文(不是逐步)。

审阅态 ≠ 文档脏

审阅过程中原文磁盘文件一直没变;合并写回才让文档变脏。 所以关掉审阅态(放弃)、切文档、突然关窗口 — 原文始终在磁盘上,产物也始终在 .typola-output/,你不会丢。

跟别家不同

Claude Desktop 用 Tab、Coze 用大列表、Notion AI 用对话侧栏 — 都太重。 Typola 学的是 OpenDesign 的 "Workspace tabs bar + Conversations pill" 双管分工,克制到极致: 下拉只做切换/新建/关闭,不做重命名/搜索/归档(那是"把对话变档案库",会引向 Claude Desktop 的聊天工作台,Typola 不干这事)。

  2. 输入框清空、Ready 状态显示

  • skill 会话 = 你从第 4 课 SkillHub 点某张 skill 卡开的会话,标题 = skill 名、UUID 独立

3. 并行 3 个会话,互不串味

操作:

  2. 点药丸 → 「+ 新建对话」开第 2 个
  3. 发 prompt 给第 2 个 — 例如「帮我看看 /huawei-style-ppt-skill 用什么场景」(skill 调试)
  4. 再开第 3 个 — 给同事写一段技术笔记标题

现在:

  • 药丸右侧出现数字徽章 3
  • 下拉面板里 3 条会话
  • 每条会话独立的 session UUID(独立 AI 上下文),互不串
验证"互不串" 的一种方法

在会话 A 里说"我是产品经理,正在写一份面向工程师的 PRD",然后切到会话 B 问"我应该用哪种语气"; AI 在 B 里不会把"产品经理 / PRD"带进来 — 它的 session 状态是干净的。 只有切同一个会话的过往对话,AI 才会接着说。

4. 关闭不用的会话

  • 点 ✕ → Typola 弹轻确认「关掉后会丢失该会话历史」
  • 确认 → 会话消失、列表少一行,药丸落到下一条活跃会话
  • 已经发给 AI 的 prompt 还在会话内存里,闭掉就没了 — 不持久化

1. 产物落到哪:固定目录,不污染工作区

Typola 强制 AI 的生成类会话(点 skill 开的、改整文档的)的 cwd 在:

<AI 工作目录>/.typola-output/<会话 ID>/

看 AI 工作目录:Composer 顶部的 WorkingDir 标签。点它能换。默认是 Typola 第一启动让你选的那个目录。

"为什么 cwd 焊这么深"

两层理由:

  • AI 写"运行时产物"(报告、PPT)→ 应该只在 .typola-output/ 出现,不会污染你原本干净的 工作目录(docs/、posts/ 等不该被随便写)
  • 保护工作目录里现有的文件 → AI 不会因为路径写错就改你仓库里的文件

想"留"某个产物 → 用「保存到工作区」(本节下面会讲);不想留 → 关 Typola / 一段时间后自动清理。

.typola-output 应该加进 .gitignore

这是临时产物,不是要入库的代码。把这一行加进你工作目录里的 .gitignore: .typola-output/

Typola 在常见工作目录里有 hint,显式不强制。

2. 右下角 chip 是产物的入口

4. 在中间编辑器打开 — 不在右栏铺预览

关键设计决策,重读三遍:

产物 = 文档。文档归文档位。

点开 chip → 中间编辑器开新 tab(像你打开任何 .md 一样):

  • .md → WYSIWYG 阅读模式,可编辑
  • .html → 走 HtmlPreviewPane 阅读模式(只读、不在中间编辑,渲染之后可点"在浏览器打开")
  • PPT HTML → 同上(只读 HTML,完美套 frontend-slides 等的输出)
不是右栏预览。右栏铺预览就是把产物当"附件 / 副产物",那是 chatbox 的设计;Typola 把产物当文档。

图示区别:

chipbox / Cursor 模式:  [文档]  |  [产物预览]    ← 文档边栏看产物,产物降级
Typola 模式:            [文档 tab:weekly-report.md]        ← 中间编辑,正常文档

5. 保存到工作区 — 长期留

在产物中心 / chip 「菜单」找「保存到工作区」:

  1. 弹目标目录选择(默认 <工作区>/ 根,可换子目录)
  2. Typola 自动改名避免冲突:已有 weekly.md → 新名 weekly-2.md(或加时间戳)
  3. 文件从 .typola-output/<会话>/ 搬运到工作区(不是复制)
  4. 产物状态从 done → archived
什么时候该"保存到工作区"
  • AI 写的报告 / 周报你要拿出来用 — 立刻归档
  • AI 写的草稿你打算改改后当终稿 — 归档后用中间编辑器编辑
  • AI 写的临时素材 / 实验产物 — 不用归档,关 Typola 后自动清理

6. 合并到当前文档(走内联审阅)

AI 写的产物 = 整篇 .md 内容?那很可能你想用它替换或合并当前编辑的草稿。

chip 旁或产物行菜单有「合并到当前文档」按钮:

  1. 触发第 5 课的内联审阅态(不是粗暴覆盖)
    • 原文 = 当前打开文档的全文
    • 新版 = 产物文件全文
  2. 进入内联审阅态后,Y/N/⏎ 逐处对比,改动地图同步亮
  3. 点「应用」→ 合并结果写回当前文档 → 文档变脏

这里的设计选择是 — 即便产物是"AI 已经写完整"的产物,也不直接覆盖原文。 你能控制每一处接受与否,跟第 5 课的产物写回走同一条管线。

"AI 全文重写"的产物也走内联审阅

有 skill output: review_inline 标签(如"全文重写"型 skill)→ 产物到达后直接进入审阅态,不必手动点合并。 这是 Typola 的 A 类入口(AI_DIFF_PREVIEW_SPEC §2.1)。

7. 覆盖原文 + 撤销覆盖

当你确定 AI 写的产物就是终稿、要直接替换当前文档(不想要审阅态),

有两种粒度:

动作 走法
覆盖当前文档 产物行点「覆盖」按钮 → 弹确认「原文将被替换,可在 5 分钟内撤销」 → 文档替换 + 产物 manifest 标记"已覆盖"
撤销覆盖 同一行点「撤销」按钮 → 文档回到覆盖前状态(产物 manifest 标记撤销)
覆盖是"少看一眼"模式 — 真要用慎重

默认用「合并到当前文档」(走内联审阅);确定完全没必要对比时再走「覆盖」。 撤销有时间窗(目前是 5 分钟),超时后撤销不可用。

Typola 怎么做

Typola 强制导出与编辑共享同一个 Markdown 源(markdownExportRenderer 是 Phase 4 共享桥接)。 4 种导出默认会把 Markdown 渲成自己的格式;导出按钮走 Rust 后端,完成后右上角 toast 通知,不打断写作。 没了 AI、没了网,Typola 仍然可以作为本地编辑器使用 — 这部分叫降级。

1. PDF 导出 — 最终交付

场景:把这篇文档原样交付 — 不再被修改、直接发出去 / 归档 / 评审。

  1. 打开要导出的文档
  2. 按 Ctrl + P 或菜单 导出 → PDF
  3. Typola 用一个隐藏的 WebView2渲染 Markdown,然后调 Page.printToPDF(本地,不上云)
  4. 保存路径自动从当前文件路径派生,无需开对话框:
    • 当前文档已存 → 同目录 / 同主名加 .pdf
    • 当前文档未存 → 默认落用户下载目录
  5. 右上角 toast:「PDF 导出成功 / 失败」
PDF 不响应你应用的主题色

Typola 的暖米黄/深海/墨韵主题只影响编辑器;PDF 走自己的导出配色(预设)。 理由:别人读 PDF 时不一定知道你用什么主题。 想自定义 PDF 样式 → 「自定义导出预设」(本节最后会展开)。

2. Word — 给会再编辑的同事

场景:把稿子给同事,他习惯改 Word — 你不用帮他去 Word 里排版。

  1. 导出 → Word (.docx)
  2. 可选:先打开Word 纸张预览(右栏的 WordPaperPreviewPane)看 Word 视角下的样子
  3. 确认后导出,产物落 <源>.docx
  4. 同事用 Word 打开可以正常修改
.docx 不是 round-trip

Typola 生成的 .docx 是「导出副本」,不是你未来再 import 回 Typola 的双向通道。 同事改完 Word 给你,你最好拿回 .md 在 Typola 里继续改 — Word 修改痕迹别强迫 Typola 反向解析。

自定义导出预设:Wor在「设置」里有 "Word 导出预设" 编辑面板,内置 8 个 slot 默认开启,你可以上传 JSON 自定义版式(字号/页边距/标题字号/字体):

  "name": "团队周报样式",
  "page": { "margin": "2.5cm", "fontSize": "10.5pt" },
  "heading": { "h1": "16pt", "h2": "14pt" },

给 AI 改
把 review.md 喂给任意 AI(Claude / GPT / Gemini / 自家 AI),它们看 blockquote 后的「请处理」知道要改什么

导出路径取决于上下文:

在哪里导出 文件名
检视模式右栏 <原文件名>.review.md(带检视版标记,不污染原 .md)
AI 改稿产物 <原 stem>.ai改{N}.md(N = 改了几次)
导出 review.md 后,原文档彻底不受影响

Typola 的检视意见是叠加在内存里的一层,导出的 .md 是这一层的快照; 关闭 Typola → 那份 .md 还在磁盘(你刚导出的),内存里的意见消失; 重新打开 .md → Typola 不会再去把意见解析回那份 .md,单向快照设计。

5. 离线降级 — 没网、没 CLI 时 Typola 还能干什么

这是 Typola 的底线承诺:所有功能必须在断网/无 AI CLI 下退化为可用。PRODUCT.md 设计原则 #4。

断网 → 你还有这些
  • 所有 Markdown 写作与编辑功能(阅读/心流/检视 三模式)
  • 本地文件操作(打开 / 保存 / 改后缀 / 拖拽)
  • 查找替换、文件大纲、文档统计、Find/Replace
  • 主题切换(本地样式)
  • 图片粘贴保存(本地)
  • WordPaper 预览 + Word/PDF 导出(本地 WebView2)
  • 底层终端(可在任意目录跑命令 — 但要 Windows 已装 terminal)
断网 → 你没有这些
  • 所有 AI 改稿(浮条、检视 → AI、AI 工作台对话)
  • SkillHub 的"未装时让 Claude 装"路径(它依赖 AI 工作)
  • 产物落盘以外的 watcher(但不关键 — 文件在磁盘就在)

6. 自定义导出预设

在「设置 → 导出预设」可以:

  • 管理 Word 预设(内置 8 个 slot 默认开启)
  • 管理 HTML 预设(同样的 slot 机制)
  • PDF 默认走标准预设,但允许改字体
预设的形态与限制
  • 是 JSON 文件,不是 LaTeX 模板 — 学起来轻
  • 设置面板里直接给实时预览
  • 8 个 slot 上限 = 防止"我做了 30 个预设然后忘"

试一试

15 分钟任务:一稿四产出
  1. 打开任意写好的文档(无 AI 也能做)。
  2. 导出 PDF 一次,在 导出物 同目录看到新 .pdf。
  3. 导出 Word 一次,确认 .docx 可用 Word/WPS 打开。
  4. 复制富文本,粘贴到一个临时文本文件,看 HTML 片段里有 <h2>/<blockquote> 等干净标签。
  5. 断网(关 wifi + 关以太网),看 Typola 是否还能写、还能导出 PDF、还能用本地终端。断网下不要去 AI 改稿,这是预期行为。

✅ 一稿四出 + 知道什么时候没 AI 也能干 = 9 节课全部完成 🎉。 从第 0 节到这里,你已经掌握 Typola 全能力:写、改、检视、交付、外加在哪一节暂停。

陷阱

  • PDF 导出卡死 — 通常是图片路径解析失败,或字体加载超时。Typola 走 fail-open 路径,等 30 秒一定出结果。
  • Word 导出排版乱 — 多半是表格/代码块超长。预设里调字号、改用更紧凑的版式。
  • 公众号粘贴后字体不动 — 公众号富文本编辑器会覆盖部分样式。你需要在后台再选一次默认字体。
  • 导出后没找到文件 — 看保存路径:已保存的文档导出到同目录,未保存的文档落到 Downloads。
  • 断网后 AI 改稿按钮变灰 — 是正常降级,不是 bug。换到无 AI 的工作流即可。
