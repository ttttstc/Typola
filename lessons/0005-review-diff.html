第 5 课

检视模式 + 内联审阅:逐处采纳 AI 改稿

15 分钟。写作者最常用的套路:切到检视模式 → 给长稿攒意见 → 一键让 AI 按意见改 → 中间编辑器逐处采纳/拒绝。

能力定位

解决什么

"我花了 3 小时写一篇稿子,现在想找同事给我提点意见" — 同事可能没空实时讲,你也不想改坏原文。 另一个真痛点:"AI 帮我按意见改了整篇,但我不知道改了哪里,怎么办?" Typola 用两层结构解决:
(1) 检视意见层 = 叠加在文档上的一层标记,不污染原文,导出前保持原稿干净。
(2) 内联审阅态 = AI 改完稿之后,中间编辑器切到只读,逐处对比,采纳/拒绝,完了再合并回原文。

Typola 怎么做

检视模式(DocumentModeSwitcher 第 3 段)进入后,选中段落仍能用第 1 课的浮条 — 这次「加检视意见」按钮有效了。 意见以「每段后展开的卡」+ 高亮被批注的文字 + 右栏汇总列表显示。 点右栏「AI 改稿」→ 发请求 → Typola 把全文+意见拼成 prompt → AI 改完产物落到 .typola-output/<会话>/xxx.ai改{N}.md → 中间编辑器整体切到内联审阅态(用自渲染只读 DOM,不被 live Vditor 擦写),显示「原文 vs 新版本」分段 diff + 顶部计数器 + 上/下一处 + Y/N/⏎ 键盘 + 右侧改动地图。 点「应用」→ 合并结果写回当前文档,退出审阅态,文档标脏。

跟别家不同

Notion 让你直接在别人的文档边输入"评论气泡";Google Docs 让你 reply 意见 —都乱入原文 DOM。 Typola 的检视意见是叠加层,导不导出 review.md 完全可控 — 原文从未污染。AI 改稿也不用 3-way merge、不用 fork UI,审阅是只读的一层,应用时一次写回 — 这避免了"实时在线编辑冲突"这个最难啃的坑。

本课目录
前置 进入检视模式 加意见(浮条「加检视意见」) 右栏汇总列表 导出 review.md 给协作者 「AI 改稿」发请求 内联审阅态:Y/N/⏎ + 改动地图 点「应用」合并回原文 试一试 陷阱

前置

  • 已完成第 1 课、第 2 课、第 4 课
  • 有一份至少 10 段以上的草稿(过短没法演示段落 diff)。
  • 装好 CLI,Typola 检测通过

1. 进入检视模式

顶栏三段控件点「检视模式」,或在阅读模式下按几次 Shift + A 配合选择。

进入后你会看到:

  • 编辑器保持当前文档不变(你没污染原文)
  • 右侧栏出现「检视意见」面板(header: 检视意见 + 数量 + 脏标记)
  • 暂时右边还要 SkillHub(可关) — 不冲突
检视模式 ≠ 检视面板常驻

检视模式是文档状态;"右栏出现检视意见"是UI 表现。关掉右栏(✕)再次打开,它还会出现 — 直到你切回阅读模式或心流模式。

2. 加意见 — 用浮条「加检视意见」

选中任意一段(或跨段选 3-5 行)→ 等浮条出现(第 1 课)→ 点「加检视意见」:

  1. 弹出检视意见编辑器浮卡(贴选区下方)
  2. 输入意见内容,例如「这段论据太单一,引用研究数据会更有说服力」
  3. 回车或点确认 → 意见入库

页面变化:

  • 被批注的段落整段高亮(主题色淡背景,不是 AR 涂红)
  • 段后展开一张意见卡:blockquote 风格 + 「**检视意见,请处理**:<你写的>」
  • 右栏「检视意见」数量 + 1
段后意见卡的格式为什么这么写

Typola 不用 emoji + 不用自定义数据属性,直接用 > **检视意见,请处理**:<意见> 这种 md blockquote 形式 — 这样导出的 review.md 本身就是干净的 Markdown,人类可读、AI 可懂。 等于一份"喂给 AI 的 prompt 格式" + 一份"给同事看的可读版"。

写 3-5 条意见,不同段落,每条意见文字不同。

3. 右栏汇总列表

右栏每个意见一条 row:

  • 上下文:被批注那段的截断文本(50 字左右)+ 「...」
  • 意见:你写的原话
  • 时间:相对时间(几分钟前 / 刚刚 / 直接日期)
  • 操作:跳转(▶) · 编辑(铅笔) · 删除(🗑)

右栏顶部还有两个全局按钮:

导出检视版
把意见合成进每段后,另存为 *.review.md(或指定后缀)
AI 改稿
切到 AI 改稿视图,审稿视图收起

4. 导出 review.md — 给协作者看

如果你今天就想把稿子丢给同事:
右栏 导出检视版 → 选保存目录 → 一份 <原文件名>.review.md 落盘。

这份文件长这样(简化):

# 你的原文章节
> **检视意见,请处理**:这段论据太单一...
段落原文

# 下一段
> **检视意见,请处理**:开头弱,缺钩子
段落原文
...

发出去,同事直接看到"哪儿有问题" — 不需要再装 Typola、不需要装 skill、不需要任何工具。

导出 review.md 后续能再打开吗?

可以,但Typola 不会再把意见解析回原文档(这是 round-trip 红线 — 在你做导出时已经走到文件,导出的 review.md 是单向快照,不是双向 sync)。 这让 review.md 永远干净,你可以心安理得把它丢给任何工具(微信、Notion、邮件都接得住)。

5. 「AI 改稿」发请求

右栏切到 AI 改稿 视图(顶部「AI 改稿」按钮)→ 你会看到:

  • 顶部「发 AI 修改」主按钮
  • 下方是「{stem}.ai改{N}.md」文件列表 — 这是 AI 历次改稿产物

点「发 AI 修改」:

  1. Typola 拼 prompt = 当前文档全文 + 所有意见(按段后格式附加)
  2. AI 工作台会话新开一个回合(不打断你现有对话)
  3. AI 改完后产物落 <工作区>/.typola-output/<会话>/<stem>.ai改1.md
  4. 中间编辑器整体切到内联审阅态(原文档隐藏,审阅 UI 接管)

6. 内联审阅态 — Y / N / ⏎ 鼠标级键盘

这是 Typola 真功夫。中间编辑器不是右栏铺开,而是直接接管。你能看到:

  • 顶部条「审阅 AI 改稿 · 第 N/总数 处 ▲▼ 全部采纳 / 全部拒绝 / 应用 ✕」
  • 左侧(占主要区域):分段后的混合视图
    → unchanged 段:直接渲染
    → modified 段:原文删除线 + 新版高亮 + 该段下方贴 ✓/✗ 按钮
    → added 段:绿色高亮 + ✓/✗
    → removed 段:删除线 + ✓(采纳删除) / ✗(保留)
  • 右侧滚动条:改动地图 — 每处一个小标记,当前处放大高亮,点就跳

键盘流(进入审阅后全程不用鼠标):

Y
采纳当前 hunk,自动跳到下一处
N
拒绝当前 hunk,自动跳到下一处
Enter / ↓
跳到下一处
Shift + Enter / ↑
跳到上一处
Esc
退出审阅态(若你动过 hunk 会有轻确认 "放弃本次 AI 审阅?")
每处默认状态 = 采纳

这是产品决策:用户最常的意图是"AI 改得还行就用",所以每处 diff 默认就是接受状态。 用户只需对不想要的改点 ✗,而不是对每处都去点击 "采纳"。 想反向操作(默认拒绝)就开 全部拒绝。

7. 点「应用」合并回原文

所有处过完一遍,点「应用」→ Typola 把当前各处状态合并成完整文本,调编辑器 setValue 写回当前文档 → 退出审阅态 → 文档此刻变脏(因为它已经被改了)。

写回是一次原子操作:一次 Ctrl + Z 能完整回到改稿前的原文(不是逐步)。

审阅态 ≠ 文档脏

审阅过程中原文磁盘文件一直没变;合并写回才让文档变脏。 所以关掉审阅态(放弃)、切文档、突然关窗口 — 原文始终在磁盘上,产物也始终在 .typola-output/,你不会丢。

试一试

15 分钟任务:完整的"检视 → AI 改 → 逐处审 → 应用"一圈
  1. 打开你的一份长稿(>10 段),切到检视模式。
  2. 选 3 段不同段落,分别用浮条「加检视意见」添 3 条意见 — 文字要明显不同(论据弱 / 标题要改 / 多一句开场)。
  3. 右栏切到「AI 改稿」,点「发 AI 修改」,等 AI 改完产物落出来。
  4. 中间编辑器切到内联审阅态,先按 Y 走过前 2 个 hunk,然后第 3 个 hunk N 拒绝。
  5. 用顶部 ▲▼ 或键盘 Enter 验证跳转,看改动地图(右侧滚动条)是否同步高亮当前位置。
  6. 点「应用」,文档改回,看是否变脏(标题栏小圆点出现),Ctrl + S 保存。
  7. Ctrl + Z 一步回到改稿前 — 验证原子撤销。

✅ 全部完成 = 进入第 6 课:多会话 Pill。 你会学到为什么 Typola 用"药丸 + 下拉"管多个 AI 会话,而不是 Tab。

陷阱

  • 点「加检视意见」卡打不开 — 当前段落没选够文字,或选区跨越多个段落被 Typola 强制重选。重新选 1 段。
  • 右栏意见不显示 — 你切到心流模式或阅读模式了。切回检视模式(顶栏三段控件)。
  • 「AI 改稿」产物始终空 — 看左栏 AI 工作台是否有错误诊断卡。多半是 CLI 鉴权过期或模型无权限,看卡上人话指引修。
  • 审阅态里点不动 ✓/✗ 按钮 — 鼠标点的就是按钮,只是有时候焦点态不太明显。用键盘 Y/N 一样的。
  • 应用写回后内容乱 — 极少见,通常是产物文本身有 markdown 头/尾噪声。先在产物上 Ctrl + O 看一眼,确认产物是干净的全文。
  • 想看 AI 改了哪但中间不切 — 没有这种模式。Typola 的设计就是:走审阅态要么全采要么全拒要么跳着选,不提供"看着原文也看着 diff"双栏。理由:Vditor 正在编辑的 DOM 会被 outerHTML 重写擦掉所有"叠加的"修改提示 — 必须把视图让给只读 DOM 才能保证准确性。
